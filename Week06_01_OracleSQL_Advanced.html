Vì sao Oracle vẫn quan trọng trong data environments

Nhiều ngân hàng, công ty bảo hiểm, tập đoàn viễn thông, và doanh nghiệp có hệ thống core lâu năm vẫn chạy Oracle làm transactional backbone. Với Data Engineer, điều này nghĩa là Oracle thường là nguồn upstream cho ETL, warehouse, reconciliation, và regulatory reporting.

Strengths và trade-offs của Oracle trong DE

  • Strengths: ổn định cao, optimizer trưởng thành, hỗ trợ tốt analytic workloads, partitioning mạnh, MERGE chuẩn enterprise.
  • Limitations: license và vận hành có thể tốn chi phí, cú pháp/behavior khác với open-source DB, cần kỷ luật tuning và thống kê dữ liệu.
✅ Oracle không phải chỉ để viết query. Với DE, Oracle là nơi bạn thiết kế logic incremental, reconciliation, và reporting có thể defend trước audit.
⚠️ Anti-pattern: xem Oracle như SQL generic rồi copy truy vấn từ internet mà không đọc execution plan, không kiểm thử rerun, không kiểm tra duplicate.

Analytic function khác GROUP BY như thế nào

GROUP BY gom nhiều dòng thành một dòng kết quả. Analytic function giữ nguyên số dòng, nhưng tính thêm metric theo cửa sổ dữ liệu. Với reporting SQL, đây là khác biệt rất quan trọng vì bạn vừa giữ chi tiết transaction vừa có tổng hợp ngay trên từng dòng.

analytic_queries.sql
SELECT
    branch_id,
    txn_date,
    txn_id,
    amount,
    ROW_NUMBER() OVER (
        PARTITION BY branch_id
        ORDER BY txn_date DESC, txn_id DESC
    ) AS rn_latest,
    RANK() OVER (
        PARTITION BY branch_id
        ORDER BY amount DESC
    ) AS amount_rank,
    DENSE_RANK() OVER (
        PARTITION BY branch_id
        ORDER BY amount DESC
    ) AS amount_dense_rank,
    LAG(amount) OVER (
        PARTITION BY branch_id
        ORDER BY txn_date
    ) AS prev_amount,
    LEAD(amount) OVER (
        PARTITION BY branch_id
        ORDER BY txn_date
    ) AS next_amount,
    SUM(amount) OVER (
        PARTITION BY branch_id
        ORDER BY txn_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total,
    AVG(amount) OVER (
        PARTITION BY branch_id
        ORDER BY txn_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS moving_avg_7d
FROM stg_branch_txn;

Latest row per key và dedup bằng ROW_NUMBER

latest_record_pattern.sql
WITH ranked AS (
    SELECT
        customer_id,
        status,
        updated_at,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id
            ORDER BY updated_at DESC, event_id DESC
        ) AS rn
    FROM cust_status_log
)
SELECT customer_id, status, updated_at
FROM ranked
WHERE rn = 1;

Top N per partition cho reporting

Ví dụ: top 3 sản phẩm doanh thu cao nhất theo từng region trong ngày. Đây là pattern phổ biến cho dashboard quản trị.

top_n_per_partition.sql
WITH ranked_sales AS (
    SELECT
        sales_date,
        region,
        product_id,
        revenue,
        ROW_NUMBER() OVER (
            PARTITION BY sales_date, region
            ORDER BY revenue DESC
        ) AS rn
    FROM fct_product_sales
)
SELECT *
FROM ranked_sales
WHERE rn <= 3;

Nếu chỉ cần top N toàn cục (không partition theo nhóm), từ Oracle 12c có thể dùng cú pháp ngắn hơn và rất dễ đọc:

top_n_global_12c.sql
SELECT product_id, revenue
FROM fct_product_sales
ORDER BY revenue DESC
FETCH FIRST 10 ROWS ONLY;
📌 Oracle hỗ trợ analytic functions mạnh và ổn định. Tuy nhiên luôn chỉ định ORDER BY rõ ràng để kết quả deterministic, nhất là khi dùng ROW_NUMBER cho dedup.
⚠️ Anti-pattern: dùng ROW_NUMBER nhưng ORDER BY thiếu tie-breaker khiến mỗi lần rerun chọn dòng khác nhau.

Bài 1: Analytic SQL cho retention reporting

Viết query tính running active users theo từng ngày và moving average 7 ngày cho mỗi channel trong bảng fct_user_activity.

analytic_retention.sql
SELECT
    activity_date,
    channel,
    active_users,
    SUM(active_users) OVER (
      PARTITION BY channel
      ORDER BY activity_date
      ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

    AVG(active_users) OVER (
      PARTITION BY channel
      ORDER BY activity_date
      ROWS BETWEEN 6 PRECEDING AND CURRENT ROW

Bài 2: Dedup batch event bằng ROW_NUMBER

GROUPING SETS: chỉ lấy mức tổng hợp bạn cần

grouping_reports.sql
SELECT
    fiscal_month,
    region,
    product_line,
    SUM(net_revenue) AS net_revenue,
    GROUPING(fiscal_month) AS g_month,
    GROUPING(region) AS g_region,
    GROUPING(product_line) AS g_product,
    GROUPING_ID(fiscal_month, region, product_line) AS grp_id
FROM fct_revenue
GROUP BY GROUPING SETS (
    (fiscal_month, region, product_line),
    (fiscal_month, region),
    (fiscal_month),
    ()
)
ORDER BY fiscal_month, region, product_line;

ROLLUP và CUBE

  • ROLLUP(a,b,c): tạo cấp (a,b,c) → (a,b) → (a) → grand total.
  • CUBE(a,b): tạo toàn bộ tổ hợp (a,b), (a), (b), ().
rollup_hierarchy_ascii.txt
Hierarchy subtotal mindset (ROLLUP):
Country -> Channel -> Detail
VN      -> Mobile  -> từng giao dịch
VN      -> Mobile  -> Subtotal Mobile
VN      -> ALL     -> Subtotal quốc gia VN
ALL     -> ALL     -> Grand Total
rollup_vs_cube.sql
-- ROLLUP: hierarchy subtotal
SELECT country, channel, SUM(amount) AS amount
FROM fct_payment
GROUP BY ROLLUP(country, channel);

-- CUBE: full combination subtotal
SELECT country, channel, SUM(amount) AS amount
FROM fct_payment
GROUP BY CUBE(country, channel);

Diễn giải subtotal rows đúng cách

Khi cột dimension bị NULL trong kết quả tổng hợp, không phải lúc nào cũng nghĩa là dữ liệu gốc NULL. Nó có thể là dòng subtotal do ROLLUP/CUBE tạo ra. Vì vậy cần nhìn thêm GROUPING() hoặc GROUPING_ID().

GROUPING(col)Ý nghĩa
0Dòng dữ liệu chi tiết hoặc col có giá trị thực
1Col đã được tổng hợp ở cấp subtotal/grand total
✅ Với báo cáo finance, GROUPING SETS giúp query ngắn hơn, dễ audit hơn, và tránh lỗi logic khi copy nhiều nhánh UNION ALL.
⚠️ Anti-pattern: đọc nhầm dòng subtotal có NULL là lỗi dữ liệu thiếu, dẫn tới xử lý sai ở downstream report.

merge_incremental.sql
MERGE INTO fct_account_txn t
USING (
    SELECT *
    FROM (
        SELECT s.*,
               ROW_NUMBER() OVER (
                 PARTITION BY txn_id
                 ORDER BY src_updated_at DESC, src_seq DESC
               ) AS rn
        FROM stg_account_txn_delta s
        WHERE src_updated_at >= :watermark_start
          AND src_updated_at <  :watermark_end
    )
    WHERE rn = 1
) s
ON (t.txn_id = s.txn_id)
WHEN MATCHED THEN
  UPDATE SET
      t.amount = s.amount,
      t.status = s.status,
      t.src_updated_at = s.src_updated_at,
      t.etl_updated_at = SYSTIMESTAMP
WHEN NOT MATCHED THEN
  INSERT (txn_id, account_id, amount, status, src_updated_at, etl_created_at, etl_updated_at)
  VALUES (s.txn_id, s.account_id, s.amount, s.status, s.src_updated_at, SYSTIMESTAMP, SYSTIMESTAMP);

Watermark và late-arriving records

Thực tế, sự kiện đến trễ là bình thường. Nên dùng sliding window, ví dụ load lại 2 ngày gần nhất để bắt bổ sung, sau đó dedup theo business key.

-- Giả sử scheduler lưu watermark_end lần chạy trước
-- Lần này mở rộng lùi 2 ngày để bắt late-arriving

Index strategy thực tế

  • B-tree: mặc định, phù hợp OLTP và lookup/range phổ biến.
  • Bitmap: hiệu quả cho cột cardinality thấp trong workload đọc nhiều, cẩn trọng với write-heavy tables.
  • Composite index: cần theo thứ tự cột sát điều kiện lọc/join thực tế.
  • Function-based index: hỗ trợ predicate có hàm, ví dụ UPPER(email).
⚠️ Bitmap index trên bảng OLTP write-heavy (insert/update liên tục) có thể gây lock contention rất nặng. Với nghiệp vụ giao dịch, ưu tiên B-tree và chỉ dùng bitmap khi workload chủ yếu là đọc dạng warehouse/reporting.

-- Khi archive/drop partition cũ, local index giúp giảm rủi ro invalid index toàn bảng.

Với bảng partitioned trong ETL, thường ưu tiên local index để khi drop/truncate partition cũ (archive/retention) không làm invalid toàn bộ index của bảng. Global index vẫn có chỗ dùng nhưng chi phí maintenance thường cao hơn khi lifecycle partition diễn ra thường xuyên.

Partition pruning mindset

Nếu query filter theo cột partition key dạng đúng chuẩn, optimizer có thể đọc ít partition hơn. Ví dụ filter theo txn_date cho tháng cụ thể giúp giảm IO rõ rệt.

Transaction basics cho ETL

BEGIN
  -- 1) Load vào staging
  -- 2) Dedup source
  -- 3) MERGE vào target
  -- 4) Validate duplicate và amount
  COMMIT;
EXCEPTION
  WHEN OTHERS THEN
    ROLLBACK;
    RAISE;
END;

Read consistency và locking cơ bản

Oracle cung cấp read consistency để reader thấy snapshot ổn định. Locking thường ở mức row-level khi update/merge. Tuy vậy, transaction quá dài hoặc batch thiết kế kém vẫn có thể gây lock contention.

📌 ETL transaction boundary nên theo đơn vị batch hợp lý. Quá nhỏ gây overhead, quá lớn gây lock kéo dài và rollback tốn kém.
⚠️ Anti-pattern: gộp toàn bộ ngày dữ liệu khổng lồ vào một transaction duy nhất mà không checkpoint, dễ gây lock contention và rollback đau đớn.

Bài 1: Partition design task

Bài 2: Transaction boundary review

Một job đang merge 40M rows trong 1 transaction và gây lock contention. Đề xuất redesign đơn giản cho người mới.

  1. Chia batch theo ngày hoặc theo partition để giảm thời gian giữ lock.
  2. Mỗi batch: dedup → merge → validate nhanh → commit.
  3. Nếu lỗi: rollback batch hiện tại, không rollback toàn bộ lịch sử.
  4. Lưu checkpoint batch_id để rerun có kiểm soát.
