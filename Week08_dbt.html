Định vị dbt trong data platform và production mindset

🎯
2. 🎯 Mục Đích Module▼
  • Phân biệt rõ dbt với ingestion tool và orchestration tool.
  • Hiểu lý do ELT phù hợp với cloud warehouse trong modern data stack.
  • Nắm đúng scope tuần 8-9 theo roadmap: models, tests, docs, snapshots, macros, commands.

📖
3. 📖 Giải Thích Chi Tiết▼

dbt là framework SQL-first cho transformation. Nó không làm Extract và không làm Load, mà tập trung vào Transform trong ELT. Nghĩa là dữ liệu được load vào warehouse trước, sau đó dbt áp dụng logic SQL theo DAG có kiểm soát.

W8.03 · Models: view / table / incremental / ephemeral
Chọn materialization đúng để tối ưu runtime và chi phí
📋
1. 📋 Menu Tổng Thể▼
BướcKết quả cần đạtThời gian ước tính
B1Nắm trade-off 4 materializations20 phút
B2Viết incremental model chuẩn25 phút
B3Kiểm tra idempotency sau 2 lần run15 phút
B4So sánh cú pháp PostgreSQL vs Oracle15 phút
🎯
2. 🎯 Mục Đích Module▼
  • Hiểu rõ khi nào dùng view/table/incremental/ephemeral.
  • Viết incremental model với is_incremental() và unique_key.
  • Biết adaptation SQL giữa PostgreSQL và Oracle cho watermark/filter.
  • Xác minh idempotency trước khi áp dụng production schedule.
  • Tránh duplicate và missing data do logic delta sai.
📖
3. 📖 Giải Thích Chi Tiết▼
MaterializationDùng khiƯu điểmLưu ý
viewstaging nhẹKhông tốn storageCó thể chậm nếu chain view dài
tablemarts truy cập nhiềuTốc độ đọc ổn địnhTốn storage, rebuild lâu
incrementaldữ liệu lớn, cập nhật theo thời gianGiảm runtime/costCần key + watermark chuẩn
ephemerallogic trung gian ngắnKhông tạo object vật lýDebug khó hơn vì compile thành CTE
fct_orders incremental
{{ config(materialized='incremental', unique_key='order_id') }}

select
  order_id,
  customer_id,
  order_status,
  order_amount,
  updated_at
from {{ ref('int_order_items_enriched') }}
{% if is_incremental() %}
where updated_at >= (select max(updated_at) from {{ this }})
{% endif %}
Watermark filter: PostgreSQL vs Oracle
-- PostgreSQL
where updated_at >= (
  select coalesce(max(updated_at), timestamp '1900-01-01 00:00:00')
  from {{ this }}
)

-- Oracle SQL
where updated_at >= (
  select nvl(max(updated_at), timestamp '1900-01-01 00:00:00')
  from {{ this }}
)
Validation query sau 2 lần run
-- PostgreSQL
select count(*) as row_count, max(updated_at) as max_updated_at
from mart.fct_orders;

-- Oracle SQL
select count(*) as row_count, max(updated_at) as max_updated_at
from MART.FCT_ORDERS;
⚠️ Nếu row_count tăng bất thường khi chạy lại cùng dữ liệu, logic incremental của bạn đang không idempotent.
❓
4. ❓ Câu Hỏi Ôn Tập▼
📌
5. 📌 Tổng Kết▼
viewTốt cho staging nhẹ và nhanh triển khai.
tableTốt cho output cuối truy vấn nhiều.
incrementalBắt buộc cho bảng lớn cập nhật liên tục.
ephemeralGiảm object trung gian không cần thiết.
Cross-DBcoalesce vs nvl, LIMIT vs FETCH FIRST.
IdempotencyLuôn test lặp để xác nhận ổn định.
🔬

W8.04 · ref() và source(): Dependency Management
Thiết kế lineage và dependency rõ ràng từ raw đến marts
📋
1. 📋 Menu Tổng Thể▼
BướcKết quả cần đạtThời gian ước tính
B1Khai báo sources trong schema.yml15 phút
B2Refactor hard-code sang ref()/source()25 phút
B3Hiểu selector +model / model+15 phút
B4Đọc lineage và impact khi đổi model15 phút
🎯
2. 🎯 Mục Đích Module▼
  • Dùng source() cho bảng raw có metadata/freshness.
  • Dùng ref() cho dependency giữa các model nội bộ.
  • Loại bỏ hard-code schema.table để tăng tính linh hoạt môi trường.
  • Hiểu tác động upstream/downstream khi model thay đổi.
  • Tăng khả năng debug và deploy theo DAG chuẩn.
📖
3. 📖 Giải Thích Chi Tiết▼

source() là điểm vào của dữ liệu raw; ref() là chuẩn tham chiếu model nội bộ. Khi dùng đúng, dbt dựng được lineage đầy đủ và đảm bảo thứ tự build hợp lý.

schema.yml cho sources
version: 2

sources:
  - name: raw
    schema: raw
    tables:
      - name: orders
      - name: customers
source() + ref()
-- models/staging/stg_orders.sql
select order_id, customer_id, status as order_status, updated_at
from {{ source('raw', 'orders') }}

-- models/marts/fct_orders.sql
select *
from {{ ref('int_order_items_enriched') }}
Dependency selector
$ dbt ls --select +fct_orders
$ dbt run --select stg_orders+
Hiển thị upstream/downstream theo DAG
✅ Không hard-code schema.table trong model. Dùng ref/source để deploy an toàn giữa dev, staging, prod.
❓
4. ❓ Câu Hỏi Ôn Tập▼
📌
5. 📌 Tổng Kết▼
source()Điểm vào chuẩn cho bảng raw.
ref()Dependency chuẩn giữa model nội bộ.
LineageTự sinh và dễ truy vết impact.
SelectorChạy đúng phạm vi +upstream/downstream.
MaintainabilityGiảm lỗi do đổi schema/tên bảng.
DeploymentDAG rõ giúp deploy có thứ tự.
🔬
6. 🔬 Bài Tập Thực Hành▶
Bài 1: Refactor hard-code SQL

Tìm ít nhất 3 model còn hard-code schema.table và chuyển hoàn toàn sang source()/ref().

Expected output: DAG hiển thị đầy đủ, dbt compile pass, không còn hard-code trong model chính.

  • Khai báo source trong YAML trước.
  • Thay tất cả from raw.xxx bằng source('raw','xxx').
  • Thay join model bằng ref('model_name').
  • Chạy dbt ls --select +marts để kiểm tra DAG.

3. 📖 Giải Thích Chi Tiết▼
Built-in tests trong schema.yml
version: 2

models:
  - name: fct_orders
    columns:
      - name: order_id
        tests: [unique, not_null]
      - name: order_status
        tests:
          - accepted_values:
              values: ['pending','paid','shipped','cancelled']
      - name: customer_id
        tests:
          - relationships:
              to: ref('dim_customers')
              field: customer_id
Custom generic test
{% test not_future_date(model, column_name) %}
select *
from {{ model }}
where {{ column_name }} > current_timestamp
{% endtest %}
Singular test
select *
from {{ ref('fct_orders') }}
where order_amount < 0
Debug test fail: PostgreSQL vs Oracle
-- PostgreSQL
select order_id, order_amount
from mart.fct_orders
where order_amount < 0
order by updated_at desc
limit 50;

-- Oracle SQL
select order_id, order_amount
from MART.FCT_ORDERS
where order_amount < 0
order by updated_at desc
fetch first 50 rows only;
⚠️ Không tắt test để pipeline xanh tạm thời. Mục tiêu là sửa root cause ở logic transform hoặc data source.

{% macro date_diff_days(start_col, end_col) %}
