Hiểu đúng bối cảnh enterprise: nhiều hệ thống rời rạc, nhiều owner, nhiều độ trễ, nhiều rủi ro nghiệp vụ.

Vì sao hệ thống doanh nghiệp bị phân mảnh?

  • CRM quản lý lead/opportunity.
  • ERP quản lý đơn hàng, công nợ, tồn kho.
  • App DB chứa hành vi user theo sản phẩm.
  • SaaS tools như marketing automation, support desk.
  • Logs/events từ ứng dụng và hạ tầng.
⚠️ Anti-pattern: tin rằng “chỉ cần export CSV mỗi ngày” là đủ. Khi volume tăng và nhiều team cùng dùng dữ liệu, cách này gây lệch số, lỗi version file và mất dấu lineage.

3. Giải Thích Chi Tiết▼

Batch integration

Batch đơn giản để vận hành, phù hợp dữ liệu kế toán và reporting theo mốc giờ/ngày. Sức mạnh: dễ backfill, dễ audit theo lô. Điểm yếu: độ trễ cao.

CDC integration

CDC đọc thay đổi từ transaction log, phù hợp đồng bộ near real-time. Bắt buộc xử lý đủ INSERT/UPDATE/DELETE và giữ offset/checkpoint đáng tin cậy.

API pull integration

Với SaaS, thường không có DB access. Bạn phải xử lý token expiry, pagination, retry theo rate-limit window, và tránh retry đồng loạt gây thundering herd.

api_ingest_loop.py (pseudo)
next_page = None
attempt = 0
base_wait_seconds = 2
max_wait_seconds = 60

while True:
  resp = call_api(page_token=next_page)

  if resp.status_code in [429, 503]:
    wait = min(max_wait_seconds, base_wait_seconds * (2 ** attempt))
    jitter = random_between(0, 1.5)
    sleep(wait + jitter)
    attempt += 1
    continue

  if resp.status_code >= 400:
    send_to_quarantine(resp.payload, reason=resp.status_code)
    break

  write_raw(resp.items)
  attempt = 0

  if not resp.next_page_token:
    break
  next_page = resp.next_page_token

Event-driven integration

Pub/Sub giúp producer và consumer tách rời. Đổi lại, bạn phải nghĩ về ordering, duplicate events và replay.

File-based integration

CSV/JSON/Parquet qua SFTP/object storage vẫn rất phổ biến khi làm việc với đối tác. Cần naming convention, manifest/checksum và rule reject file lỗi.

⚠️ Anti-pattern: Use case báo cáo doanh thu D+1 nhưng lại ép realtime CDC + event stream, gây tăng chi phí, tăng complexity và khó vận hành không cần thiết.

Incremental watermark query (an toàn với late-arriving data)

Nếu chỉ dùng updated_at > last_success_ts, bạn có thể mất record commit trễ (late arriving). Production thường dùng overlap window rồi dedup ở downstream.

PostgreSQL
Oracle SQL
batch_incremental.sql
SELECT order_id, updated_at, status
FROM orders
WHERE updated_at >= :last_success_ts - INTERVAL '1 hour'
ORDER BY updated_at ASC
LIMIT 100000;
batch_incremental.sql
SELECT order_id, updated_at, status
FROM orders
WHERE updated_at >= :last_success_ts - INTERVAL '1' HOUR
ORDER BY updated_at ASC
FETCH FIRST 100000 ROWS ONLY;
📌 Nếu nguồn có hard delete, batch incremental theo watermark sẽ không thấy record đã biến mất. Cần CDC với tombstone (op_type='D') hoặc full snapshot + checksum để phát hiện delete.

Identity mapping (xương sống của Customer 360)

Khi tích hợp CRM, app và payment, cùng một khách hàng thường có nhiều ID khác nhau. Cần bảng map định danh để nối đúng thực thể, tránh cộng dồn sai doanh thu hoặc tính trùng người dùng.

identity_resolution.sql
SELECT c.crm_contact_id,
       a.app_user_id,
       p.payment_customer_id,
       COALESCE(idm.global_customer_key, md5(lower(c.email))) AS global_customer_key
FROM crm_contacts c
LEFT JOIN app_users a
  ON lower(c.email) = lower(a.email)
LEFT JOIN payment_customers p
  ON regexp_replace(c.phone, '[^0-9]', '', 'g') = p.phone_normalized
LEFT JOIN identity_map idm
  ON idm.source_system = 'crm'
 AND idm.source_id = c.crm_contact_id;

Schema evolution: additive vs breaking

  • Additive change: thêm cột mới optional, consumer cũ vẫn chạy.
  • Breaking change: đổi tên cột, đổi type incompatible, đổi semantic enum.
⚠️ Anti-pattern: producer thêm/đổi cột âm thầm rồi đẩy vào production, khiến pipeline downstream fail sau nhiều giờ.

Idempotency trong integration là gì?

Một record được xử lý lại nhiều lần nhưng target state vẫn đúng như xử lý một lần. Thực tế thường dùng natural key + event_time/version hoặc idempotency key.

Edge case khó: out-of-order events (event update đến trước event create). Guardrail thực tế là so sánh version/event_time khi upsert, chỉ chấp nhận bản ghi mới hơn và giữ track trạng thái chờ nếu thiếu bản ghi gốc.

Replay-safe pipeline

replay_safe_flow.md
1) Read batch/events
2) Validate + normalize
3) Upsert by business key
4) Commit checkpoint ONLY after target write success
5) Emit reconciliation metrics

Retries: transient vs fatal

Loại lỗiVí dụCách xử lý
TransientHTTP 429/503, DB connection timeoutRetry + backoff + jitter
FatalSchema mismatch, contract violationFail-fast + alert + không retry vô hạn

Partial file load & poison message

File có 1.000.000 record, 100 record lỗi format: không nên drop cả file hoặc nuốt lỗi. Cách chuẩn: route record lỗi vào dead_letter_queue để xử lý sau, phần hợp lệ vẫn đi tiếp nếu policy cho phép.

Backpressure khi consumer xử lý không kịp

Nếu tốc độ ingest vượt tốc độ xử lý downstream, queue lag sẽ tăng và SLA vỡ dây chuyền. Cần cơ chế pause/resume, giới hạn concurrency, và alert theo consumer lag thay vì chỉ nhìn job status.

failure_flow.md (ASCII)
API Source
   |
   v
Ingestion Worker --(429/503)--> Retry Queue (exponential backoff + jitter)
   |
   +--(400 invalid payload)--> Quarantine/DLQ --> Alert on-call --> Fix mapping/contract
   |
   +--(success)--> Raw/Staging --> Idempotent Upsert --> Checkpoint Commit
📌 Exactly-once “tuyệt đối” hiếm khi có trong hệ thống phân tán end-to-end. Thực tế production thường là at-least-once + idempotent writes + reconciliation.
⚠️ Anti-pattern: rerun job sau lỗi nhưng append toàn bộ vào bảng đích, tạo duplicate âm thầm khiến báo cáo phình số.

INSERT INTO fct_order_status (order_id, status, event_time, source_event_id)
SELECT order_id, status, event_time, source_event_id
FROM stg_order_status

idempotent_upsert.sql
MERGE INTO fct_order_status t
USING stg_order_status s
ON (t.source_event_id = s.source_event_id)
WHEN MATCHED THEN UPDATE SET
  t.status = s.status,
  t.event_time = s.event_time
  WHERE t.event_time <= s.event_time
WHEN NOT MATCHED THEN
  INSERT (order_id, status, event_time, source_event_id)
  VALUES (s.order_id, s.status, s.event_time, s.source_event_id);

Reconciliation tối thiểu

api_ingestion_with_backoff.py (rút gọn)
import random
import time

def call_with_retry(fetch_page, page_token):
    attempt = 0
    while attempt < 8:
        resp = fetch_page(page_token)
        if resp.status_code in (429, 503):
            wait = min(60, (2 ** attempt)) + random.uniform(0, 1.2)
            time.sleep(wait)
            attempt += 1
            continue
        if resp.status_code >= 400:
            raise RuntimeError(f"fatal api error: {resp.status_code}")
        return resp
    raise RuntimeError("retry exhausted")

# checkpoint chỉ commit sau khi write thành công
# để tránh mất dữ liệu khi job dừng giữa chừng

SELECT ds,

       target_total_amount,
       CASE
         WHEN source_cnt = target_cnt
          AND source_total_amount = target_total_amount THEN 'PASS'
         ELSE 'FAIL'
       END AS reconciliation_status

ORDER BY ds DESC;
