W13.02 – Amazon S3 cho Data Lake: raw / staging / curated / archive
Module 2/11 · thiết kế data lake thực tế thay vì ví dụ toy.
1
Menu Tổng Thể▼
PhầnNội dungMục tiêu
Zone strategyraw/staging/curated/archiveKiểm soát vòng đời dữ liệu
Naming & partitionbucket/prefix/domain/source/dt/hourDễ query, dễ governance
Lifecycle & securityStorage class, encryption, block public accessTối ưu cost + giảm rủi ro
2
Mục Đích Module▼
  • Thiết kế S3 data lake có cấu trúc đủ dùng cho batch lẫn reprocessing.
  • Áp dụng naming convention nhất quán theo domain và môi trường.
  • Tránh partition anti-pattern gây chi phí scan cao ở Athena.
  • Đặt security defaults cho bucket từ đầu dự án.
3
Giải Thích Chi Tiết▼

S3 là object storage trung tâm của nhiều pipeline DE trên AWS. Thiết kế tốt không chỉ là tạo bucket, mà là chuẩn hóa zone, partition, file size, retention và access boundaries để Athena/Glue chạy ổn định khi dữ liệu tăng mạnh.

1) Zone design thực dụng

ZoneDữ liệu chứaRetention gợi ýNgười dùng chính
rawBản gốc từ source, chưa biến đổi90-365 ngàyIngestion + forensics
stagingDữ liệu đã chuẩn schema bước đầu30-90 ngàyTransform jobs
curatedDataset phục vụ BI/ML đã kiểm thửtheo policy sản phẩmAnalytics/serving
archiveDữ liệu lâu năm, truy cập hiếmnhiều nămCompliance/audit

2) Naming, partition và Small File Problem

S3 key convention
s3://de-learning-raw-ap-southeast-1-prod/
  domain=commerce/source=erp/entity=orders/
  dt=2026-04-22/hour=13/part-0001.parquet

Partition theo dt và hour thường cân bằng giữa tốc độ query và số lượng file. Không nên partition theo user_id hoặc transaction_id vì cardinality quá cao.

Small File Problem: hàng triệu file vài KB/MB khiến Athena phải mở rất nhiều object, tăng metadata overhead, tăng thời gian lập kế hoạch query và scan cost không cần thiết.

compaction guideline
Mục tiêu file Parquet sau compaction:
- Workload vừa: 128MB - 256MB/file
- Workload lớn: 256MB - 512MB/file

Pattern:
raw nhiều file nhỏ -> job compaction theo dt/hour -> curated file lớn ổn định

3) Lifecycle, Intelligent-Tiering và network path

lifecycle policy production-minded
{
  "Rules": [
    {
      "ID": "raw-tiering-and-cleanup",
      "Filter": {"Prefix": "domain=commerce/"},
      "Status": "Enabled",
      "Transitions": [
        {"Days": 30, "StorageClass": "INTELLIGENT_TIERING"}
      ],
      "AbortIncompleteMultipartUpload": {"DaysAfterInitiation": 7},
      "Expiration": {"Days": 365}
    }
  ]
}

Dùng S3 VPC Gateway Endpoint cho EC2/Airflow/ETL trong private subnet giúp traffic tới S3 đi nội bộ AWS, tránh đi qua NAT Gateway không cần thiết.

Ghi chú S3 data lake tốt phải có: versioning, block public access, SSE-KMS, tag owner/cost-center, lifecycle rõ theo zone, compaction định kỳ.
Lưu ý Anti-pattern tốn tiền: mọi job private subnet đọc/ghi S3 qua NAT Gateway. Với lưu lượng lớn, NAT data processing fee tăng nhanh và khó phát hiện sớm.

3
Giải Thích Chi Tiết▼

Role theo workload giúp giảm rủi ro lan rộng khi một job bị lỗi hoặc bị lộ credential. Trong thực tế, ingestion job không cần quyền đọc curated, và BI query role không nên có quyền ghi raw.

Role split gợi ý

RoleQuyền cần cóKhông nên có
de-ingest-rolePutObject vào raw prefix, GetObject từ source landingDeleteObject toàn bucket
de-transform-roleRead raw/staging, write curated, Glue update tableIAM policy modify
de-analytics-roleAthena query, read curated, read Glue catalogWrite raw/staging

sts:AssumeRole và Trust Relationship

trust_policy_data_access_role.json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/airflow-ec2-role"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

Pipeline thực tế: Airflow chạy trên EC2 bằng airflow-ec2-role, mỗi DAG task gọi STS để AssumeRole sang de-data-access-role, nhận temporary credentials theo session ngắn, rồi mới đọc/ghi đúng prefix S3.

IAM policy JSON thực tế (ingestion)

iam_policy_ingest.json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowListSpecificPrefix",
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::de-learning-raw-ap-southeast-1-prod",
      "Condition": {
        "StringLike": {
          "s3:prefix": ["domain=commerce/source=erp/*"]
        }
      }
    },
    {
      "Sid": "AllowPutOnlyToIngressPath",
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:AbortMultipartUpload"],
      "Resource": "arn:aws:s3:::de-learning-raw-ap-southeast-1-prod/domain=commerce/source=erp/*"
    }
  ]
}
Tip Temporary credentials qua STS an toàn hơn access key dài hạn: có thời hạn, xoay vòng tự động, giảm blast radius nếu lộ thông tin.
Lưu ý Anti-pattern nghiêm trọng: lưu access key tĩnh trong Airflow variable hoặc file cấu hình. Khi key bị lộ, kẻ tấn công có thể dùng lâu dài nếu không rotate kịp.

Mục Đích Module▼
  • Phân biệt khi nào chọn RDS PostgreSQL, khi nào nâng lên Aurora.
  • Thiết kế schema metadata phục vụ observability pipeline.
  • Viết query vận hành cho cả PostgreSQL và Oracle SQL.
  • Xác định backup, retention, và chỉ số monitor quan trọng.

Giải Thích Chi Tiết▼

Metadata DB không chứa dữ liệu fact lớn mà chứa trạng thái chạy job, row counts, thời gian bắt đầu/kết thúc, error code. Đây là lớp dữ liệu quyết định khả năng debug và vận hành.

Khi số ETL jobs tăng, vấn đề thường gặp là connection exhaustion: mỗi job mở nhiều session, DB chạm giới hạn kết nối, query treo dây chuyền và pipeline fail dù CPU chưa cao.

RDS hay Aurora?

Tiêu chíRDS PostgreSQLAurora PostgreSQL-Compatible
Chi phí baselineThấp hơnCao hơn
HA/failoverTốt cho nhu cầu vừaTốt hơn khi SLA nghiêm ngặt
Read scalingGiới hạn hơnHỗ trợ read replicas linh hoạt hơn
Khi phù hợpWorkload metadata vừa, team nhỏMulti-team, tải cao, yêu cầu failover nhanh

Schema metadata mẫu

pipeline_runs schema
run_id, pipeline_name, run_date, status,
started_at, ended_at, processed_rows,
failed_step, error_code, retry_count

Truy vấn vận hành (PostgreSQL / Oracle SQL)

postgresql
SELECT pipeline_name,
       run_date,
       status,
       processed_rows,
       ended_at - started_at AS duration
FROM pipeline_runs
WHERE run_date >= CURRENT_DATE - INTERVAL '7 day'
ORDER BY run_date DESC, pipeline_name;
oracle sql
SELECT pipeline_name,
       run_date,
       status,
       processed_rows,
       (ended_at - started_at) * 24 * 60 AS duration_minutes
FROM pipeline_runs
WHERE run_date >= TRUNC(SYSDATE) - 7
ORDER BY run_date DESC, pipeline_name;

Kết nối DB ở production: RDS Proxy / PgBouncer

Giải phápKhi phù hợpLợi ích chính
RDS ProxyWorkload AWS-native, muốn managed poolingGiảm connection churn, failover mượt hơn cho client
PgBouncerCần kiểm soát pooling chi tiết ở PostgreSQLGiảm số session thật vào DB, ổn định khi burst job

Nguyên tắc: nhiều job ETL chỉ nên mượn connection ngắn hạn qua pool, không giữ session mở quá lâu theo vòng đời DAG/task.

Ghi chú Ít nhất phải monitor: CPU utilization, free storage, active connections, failover events.

Athena external table mẫu mạnh hơn

athena_external_table.sql
CREATE EXTERNAL TABLE IF NOT EXISTS raw_orders (
  order_id string,
  customer_id string,
  order_ts timestamp,
  total_amount decimal(18,2),
  payment_status string
)
PARTITIONED BY (
  dt string,
  source string
)
STORED AS PARQUET
LOCATION 's3://de-learning-raw-ap-southeast-1-prod/domain=commerce/entity=orders/'
TBLPROPERTIES ('parquet.compression'='SNAPPY');

MSCK REPAIR TABLE raw_orders;

Query pattern tối ưu cost cho Athena

athena_query_pattern.sql
SELECT dt,
       payment_status,
       SUM(total_amount) AS gmv
FROM raw_orders
WHERE dt BETWEEN '2026-04-01' AND '2026-04-07'
  AND source = 'erp'
GROUP BY dt, payment_status
ORDER BY dt, payment_status;
Tip Với Athena, tối ưu lớn nhất thường đến từ: Parquet + partition pruning + tránh SELECT * + workgroup có bytes scan quota.
Ghi chú Cost/performance trade-off: Athena rẻ khi query không đều; Redshift/Redshift Serverless thường hiệu quả hơn cho BI truy cập dày và SLA latency rõ.
Lưu ý Anti-pattern: dùng Athena cho dashboard realtime nhiều người dùng cùng lúc và kỳ vọng độ trễ ổn định như warehouse chuyên dụng.

Khung 4 câu hỏi trước khi chọn dịch vụ

  1. Workload chính là gì: ad-hoc, scheduled BI, hay near real-time?
  2. SLA/latency yêu cầu bao nhiêu và có thay đổi theo mùa vụ không?

Bảng chi phí tham chiếu theo thành phần

Thành phầnDriver chi phí chínhCách tối ưu thực dụng

RedshiftCompute giờ chạyReserved capacity, scaling theo giờ cao điểm

NAT Gateway hidden bill và placement theo AZ/region

  • Workload private subnet đọc S3 qua NAT liên tục sẽ tạo khoản phí lớn theo GB, kể cả khi compute không tăng.
  • Dùng S3 VPC Endpoint giảm luồng qua NAT cho traffic S3 nội bộ.
  • Đặt dịch vụ cùng region và ưu tiên cùng AZ khi phù hợp giúp giảm data transfer cross-AZ/cross-region không cần thiết.

Security baseline tuần 13

  • S3 Block Public Access bật toàn cục.
  • Encryption at rest: SSE-KMS cho bucket chứa dữ liệu nghiệp vụ.
  • Least privilege cho role pipeline theo prefix cụ thể.
  • CloudTrail/CloudWatch logs bật để audit thao tác nhạy cảm.
  • Không lưu secret trong code, dùng Secrets Manager hoặc Parameter Store tùy mức nhạy cảm và nhu cầu rotation.
