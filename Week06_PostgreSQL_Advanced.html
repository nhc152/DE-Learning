3. Giải Thích Chi Tiết ▼

PostgreSQL trong hệ Data Engineering

  • OLTP source: nơi dữ liệu gốc phát sinh từ ứng dụng.
  • Staging DB: nhận raw batch/CDC trước khi đẩy vào DWH.
  • CDC source: phát change event từ WAL qua Logical Replication (thường đi vào Debezium/Kafka trước khi vào lake/warehouse).
  • Metadata DB: lưu run history, config, quality check log.
  • Analytics-lite: dashboard nhỏ, ad-hoc query cho team nội bộ.

Khi nào PostgreSQL chưa đủ

  • Concurrency analytics cao + query quét bảng lớn liên tục.
  • Storage vài chục TB và retention dài nhiều năm.
  • Cần tách compute/storage độc lập, auto-scale mạnh cho BI workload.
Mindset nhanh khi chọn nền tảng
Nếu workload chính là app transaction + ETL vừa phải:
PostgreSQL đủ tốt, chi phí thấp, đơn giản vận hành.

Nếu workload chính là BI quét dữ liệu lớn liên tục:
Cân nhắc warehouse (BigQuery/Snowflake/Redshift) sớm.
⚠️ Anti-pattern: dùng PostgreSQL làm warehouse lớn nhưng vẫn giữ index/partition theo kiểu tạm bợ, dẫn đến chậm dần theo thời gian.

⚡ Indexing Strategy & Query Performance
B-tree · GIN · BRIN · composite/partial/expression index
📋
1. Menu Tổng Thể▼
Chủ đềÝ chínhLưu ý production
B-treeIndex mặc định cho equality/rangeHợp với cột selectivity cao
GINTốt cho JSONB, array, full-textWrite cost cao hơn B-tree
BRINNhẹ cho bảng rất lớn, dữ liệu có thứ tựHiệu quả với cột thời gian append-only
CompositeTối ưu nhiều cột theo thứ tựColumn order rất quan trọng
PartialIndex subset dữ liệuGiảm size index, tăng hiệu quả
ExpressionIndex theo biểu thứcTránh function khiến mất index usage
🎯
2. Mục Đích Module▼
  • Biết chọn index theo truy vấn thực tế, không tạo index theo cảm giác.
  • Hiểu trade-off: read nhanh hơn nhưng write chậm hơn.
  • Tránh over-indexing làm ETL insert/update chậm dần.
📖
3. Giải Thích Chi Tiết▼

1) Selectivity quyết định sức mạnh index

Nếu cột có ít giá trị lặp (ví dụ country_code chỉ vài chục giá trị), index thường ít hiệu quả hơn cột có nhiều giá trị khác nhau.

2) Composite index phải bám pattern WHERE + ORDER BY

Ví dụ index theo truy vấn thật
-- Query thường dùng:
SELECT *
FROM events
WHERE tenant_id = 'acme'
  AND event_time >= NOW() - INTERVAL '7 days'
ORDER BY event_time DESC
LIMIT 100;

-- Index phù hợp:
CREATE INDEX idx_events_tenant_time
  ON events (tenant_id, event_time DESC);

3) Partial index giảm chi phí

Với bảng jobs, nếu chỉ query job chưa xong thì index partial sẽ nhỏ và nhanh hơn index full table.

Partial index
CREATE INDEX idx_jobs_pending
  ON jobs (scheduled_at)
  WHERE status IN ('queued', 'retry');

4) BRIN cho bảng time-series append-only

BRIN (Block Range Index) không index từng row như B-tree mà lưu metadata theo block range. Vì vậy index rất nhỏ, tạo nhanh, phù hợp bảng log/event lớn có dữ liệu tăng theo thời gian.

BRIN cho event_time
CREATE INDEX idx_events_event_time_brin
  ON events USING BRIN (event_time);

-- Hợp khi dữ liệu insert gần theo thứ tự thời gian.
-- Nếu dữ liệu random mạnh, BRIN có thể kém hiệu quả hơn B-tree.
⚠️ Anti-pattern: tạo index cho mọi cột "để chắc". Kết quả thường là write latency tăng và autovacuum nặng hơn.
❓

🔎 EXPLAIN ANALYZE & Execution Plans
Planner vs Executor · đọc node quan trọng · estimated vs actual
📋
1. Menu Tổng Thể▼
Khái niệmMô tảỨng dụng
PlannerChọn chiến lược chạy queryHiểu vì sao query chọn Seq Scan hay Index Scan
ExecutorThực thi plan đã chọnQuan sát thời gian thực và rows thực tế
EXPLAINXem kế hoạch ước lượngKiểm tra nhanh trước khi chạy nặng

🎯
2. Mục Đích Module▼
  • Đọc được các node phổ biến: Seq Scan, Index Scan, Bitmap Heap Scan, Hash Join.
  • Nhìn chênh lệch estimated rows vs actual rows để phát hiện stats cũ.
  • Dùng pg_stat_statements để ưu tiên đúng query chậm trước khi EXPLAIN.
  • Tối ưu theo bằng chứng plan thay vì thử-may-rủi.
📖
3. Giải Thích Chi Tiết▼

Ví dụ plan trước khi tối ưu

EXPLAIN ANALYZE trước index
EXPLAIN ANALYZE
SELECT *
FROM orders
WHERE customer_id = 9123
  AND created_at >= NOW() - INTERVAL '30 days';

Seq Scan on orders  (cost=0.00..183421.10 rows=141 width=128)
  Filter: ((customer_id = 9123) AND (created_at >= ...))
  Rows Removed by Filter: 4231192
Execution Time: 842.331 ms

Sau khi thêm composite index

EXPLAIN ANALYZE sau index
CREATE INDEX idx_orders_customer_created
  ON orders (customer_id, created_at DESC);

Index Scan using idx_orders_customer_created on orders
  (cost=0.43..102.54 rows=148 width=128)
Execution Time: 8.943 ms

Estimated rows lệch lớn nghĩa là gì?

Nếu estimated 100 rows nhưng actual 500,000 rows, planner dễ chọn kế hoạch sai. Lúc này cần ANALYZE hoặc xem lại skew dữ liệu.

Tìm query chậm trước khi EXPLAIN bằng pg_stat_statements

Trong production có hàng ngàn query, bạn nên khoanh vùng top query tốn thời gian nhất trước, rồi mới chạy EXPLAIN ANALYZE để tối ưu đúng điểm đau.

Top slow queries (mean time)
SELECT
  calls,
  ROUND(total_exec_time::numeric, 2) AS total_ms,
  ROUND(mean_exec_time::numeric, 2) AS mean_ms,
  rows,
  LEFT(query, 120) AS query_sample
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
🧠 Quy tắc đơn giản: tìm top query chậm bằng pg_stat_statements, đọc node tốn thời gian nhất trong plan, rồi mới tối ưu theo bằng chứng.

3. Giải Thích Chi Tiết▼

Range partition cho bảng events theo tháng

Partition theo thời gian
CREATE TABLE events_p (
  id BIGSERIAL,
  event_time TIMESTAMP NOT NULL,
  tenant_id TEXT NOT NULL,
  payload JSONB
) PARTITION BY RANGE (event_time);

CREATE TABLE events_2026_04 PARTITION OF events_p
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

CREATE TABLE events_2026_05 PARTITION OF events_p
  FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');

Partition pruning

Khi query có điều kiện event_time, planner bỏ qua các partition không liên quan. Đây là lợi ích chính, không phải "tự động nhanh mọi query".

Retention

Retention theo partition
-- Ví dụ giữ 6 tháng:
ALTER TABLE events_p DETACH PARTITION events_2025_10;
DROP TABLE events_2025_10;
⚠️ Partitioning không thay thế indexing. Nếu query lọc tenant_id nhưng không có index phù hợp, vẫn có thể chậm.

📋

2. Mục Đích Module▼

  • Biết vì sao transaction kéo dài làm bloat và chặn vacuum.
  • Thiết kế ETL theo nguyên tắc: ngắn, rõ ràng, idempotent.
📖
3. Giải Thích Chi Tiết▼

MVCC ngắn gọn

PostgreSQL lưu nhiều phiên bản row. Reader thường đọc snapshot phù hợp mà không bị writer chặn trực tiếp.

MVCC timeline trực quan
T1 (09:00): BEGIN; SELECT balance FROM wallet WHERE user_id = 42;  -- thấy 100
T2 (09:01): BEGIN; UPDATE wallet SET balance = 70 WHERE user_id = 42; COMMIT;
T1 (09:02): SELECT lại trong cùng transaction vẫn thấy 100 (snapshot cũ)
T1 (09:03): COMMIT; -- transaction mới sau đó sẽ thấy 70

Lock contention thường gặp trong ETL

  • Job A giữ transaction quá lâu trong khi Job B cần update cùng rows.
  • Batch update không có điều kiện tốt dẫn đến lock lan rộng.
  • Thứ tự update bảng không nhất quán giữa các job gây deadlock.
Ví dụ transaction rõ ràng
BEGIN;

UPDATE inventory
SET reserved_qty = reserved_qty + 1
WHERE sku = 'SKU-42';

INSERT INTO reservation_log (sku, reserved_at)
VALUES ('SKU-42', NOW());

COMMIT;

Runtime troubleshooting với pg_stat_activity

Khi hệ thống bị treo lock, đừng đoán. Hãy xác định session nào đang block hoặc idle in transaction để xử lý đúng tiến trình.

Find blocker / idle transaction
-- Session đang idle in transaction (thường gây giữ lock lâu):
SELECT
  pid,
  usename,
  state,
  now() - xact_start AS xact_age,
  wait_event_type,
  wait_event,
  LEFT(query, 120) AS query_sample
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY xact_start;

-- Ai đang block ai:
SELECT
  a.pid AS blocked_pid,
  pg_blocking_pids(a.pid) AS blocking_pids,
  LEFT(a.query, 120) AS blocked_query
FROM pg_stat_activity a
WHERE array_length(pg_blocking_pids(a.pid), 1) > 0;

-- Chỉ terminate khi đã xác nhận đúng session thủ phạm:
-- SELECT pg_terminate_backend(<pid>);
⚠️ Anti-pattern: mở transaction rồi gọi API ngoài hoặc xử lý file lớn bên trong transaction. Hãy tách phần I/O ra ngoài.

3. Giải Thích Chi Tiết▼

Dead tuples hình thành thế nào?

Trong MVCC, UPDATE không ghi đè row cũ mà tạo row mới. Row cũ thành dead tuple và cần vacuum dọn dẹp.

HOT update và fillfactor cho bảng update-heavy

Với bảng staging/fact bị UPDATE nhiều, HOT (Heap-Only Tuple) giúp giảm ghi thêm vào index nếu update không đụng cột index. Để tăng cơ hội HOT update, thường giảm fillfactor để chừa chỗ trống trên page.

Set fillfactor cho bảng update-heavy
ALTER TABLE fact_order_status
  SET (fillfactor = 90);

-- Sau đó cần rewrite table (ví dụ VACUUM FULL / CLUSTER / pg_repack)
-- để fillfactor mới áp dụng toàn bộ dữ liệu cũ.

Lệnh bảo trì cơ bản

Maintenance command
VACUUM events;
VACUUM ANALYZE events;
-- Dùng cẩn thận vì lock mạnh:
VACUUM FULL events;

Kiểm tra bảng có nguy cơ bloat

Monitoring đơn giản
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  last_vacuum,
  last_autovacuum,
  last_analyze,
  last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
⚠️ Anti-pattern nguy hiểm: tắt autovacuum vì "đỡ tốn CPU" nhưng không có lịch maintenance thay thế.

3. Giải Thích Chi Tiết▼

Backup logical cơ bản

pg_dump / pg_restore
-- Backup dạng custom:
pg_dump -h localhost -U de_user -d de_lab -Fc -f de_lab_2026_04_22.dump

-- Restore vào DB mới:
createdb -h localhost -U de_user de_lab_restore
pg_restore -h localhost -U de_user -d de_lab_restore de_lab_2026_04_22.dump

Connection management

  • Không tăng max_connections vô tội vạ.
  • Dùng pooler (ví dụ PgBouncer) để gom kết nối ngắn hạn.
  • Đóng idle connection từ app đúng cách.

Bẫy PgBouncer: session pooling vs transaction pooling

Transaction pooling thường phù hợp cho workload ETL/app ngắn-lệnh vì trả connection nhanh về pool sau mỗi transaction. Nếu vô tình giữ workflow theo kiểu session state dài, bạn dễ gặp lỗi khó đoán hoặc pool cạn.

⚠️ Với Airflow hoặc worker chạy nhiều task song song, dùng session-level pooling sai cách có thể giữ connection quá lâu và làm nghẽn pool toàn hệ thống.

Checklist vận hành tối thiểu

  • Có lịch backup rõ tần suất (daily/weekly).
  • Có restore drill ít nhất hàng tháng.
  • Có theo dõi số kết nối active vs idle.
  • Có rotation credential theo chu kỳ.
🛡️ Backup strategy nên đi kèm mục tiêu RPO/RTO rõ ràng để chọn tần suất và quy trình khôi phục phù hợp.
