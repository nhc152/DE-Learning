Tổng Quan: Vì Sao Cần Apache Spark?

📌 Spark là hệ thống distributed data processing. Giá trị của Spark nằm ở quyết định kiến trúc và tối ưu chi phí chạy job, không nằm ở việc gọi nhiều API.
3
3. Giải Thích Chi Tiết▼

1) Spark sinh ra để xử lý workload nào?

Khi dữ liệu vượt khả năng RAM của một máy hoặc khi join/aggregation có chi phí quá lớn trên một node đơn, bạn cần xử lý phân tán. Spark chia công việc ra nhiều executor để chạy song song và gom kết quả.

2) Khi nào pandas / SQL đơn thuần không đủ?

Tình huốngDấu hiệuHướng phù hợp
Pandas localOOM khi đọc file lớn, runtime hàng giờChuyển sang Spark DataFrame
Warehouse SQLChi phí query ad-hoc quá cao cho backfill lặp lạiDùng Spark job có logic tối ưu rõ
Backfill lịch sửCần chạy theo partition 2-3 năm dữ liệuSpark + partition-aware writes

3) Spark trong hệ sinh thái DE

  • Airflow orchestration: khi nào chạy Spark job, retry, SLA.
  • dbt modeling ở warehouse: transform SQL có test trong lớp curated.
  • Spark xử lý nặng: historical backfill, large joins, pre-aggregation.
  • Warehouse phục vụ BI/query tiêu thụ downstream.
Pipeline mindset
# Spark không thay thế toàn bộ stack
Raw ingest → Spark (heavy transform/backfill) → Curated table → dbt marts → BI
# Airflow điều phối, Spark xử lý nặng, dbt chuẩn hóa semantic layer

4) Khi Spark là overkill

  • Dữ liệu vài trăm MB và workload chạy 1-2 phút trong SQL/pandas.
  • Team chưa có monitoring/logging/runbook, nhưng cố chạy cluster lớn.
  • Job chỉ làm filter đơn giản, không cần distributed compute.
⚠️ Anti-pattern: dùng Spark chỉ vì “big data trend”. Kết quả thường là chi phí cluster tăng nhưng latency/độ ổn định không cải thiện.

5) Week outcomes checklist

  • Giải thích được Driver/Executor/Stage/Task/Shuffle bằng ví dụ thật.
  • Ra quyết định join strategy theo kích thước data thay vì cảm tính.
  • Đọc Spark UI để tìm bottleneck thay vì đoán mò.
  • Có artifact spark_jobs/ với test chạy được.

Spark Architecture: Driver, Executor, Stage, Task, Shuffle
Hiểu cách job đi qua DAG và vì sao shuffle là điểm đắt đỏ nhất
1
1. Menu Tổng Thể▼
Thành phầnVai tròRủi ro nếu hiểu sai
DriverLập plan, điều phối taskDriver OOM khi collect dữ liệu lớn
ExecutorChạy task, giữ cache/shuffle dataMemory pressure, spill ra disk
Stage/TaskĐơn vị thực thi theo DAGTuning mù nếu không đọc stage boundary
ShuffleRedistribute dữ liệu qua networkNetwork + disk I/O tăng mạnh
2
2. Mục Đích Module▼
  • Nắm được luồng chạy Spark job ở mức hệ thống.
  • Phân biệt narrow transformation và wide transformation bằng trực giác dữ liệu di chuyển.
  • Biết vì sao shuffle đắt: memory + network + disk.
3
3. Giải Thích Chi Tiết▼

1) Cách job chạy ở mức cao

Execution flow
Driver nhận code → build logical plan → DAG scheduler tách stages
→ Task scheduler gửi tasks cho executors → executors chạy và trả result/metrics

2) DAG planning và stage boundary

Transformations như select, filter thường là narrow. Các bước như groupBy, join (không broadcast) tạo wide dependency, gây shuffle và mở stage mới.

3) Vì sao shuffle đắt?

  • Dữ liệu phải serialize, truyền qua network giữa executors.
  • Intermediate data có thể spill xuống disk khi thiếu memory.
  • Tạo nhiều file shuffle nhỏ gây overhead đọc/ghi lớn.
Chi phíNguồn gốcTác động
MemorySort/aggregation bufferGC tăng, spill
NetworkExchange dữ liệu giữa partitionsStage kéo dài
DiskShuffle spill filesI/O bottleneck

4) Driver OOM vs Executor OOM (phân biệt để xử lý đúng)

Loại OOMDấu hiệu thường gặpNguyên nhân hay gặpHướng xử lý thực tế
Driver OOMJob fail khi collect()/toPandas() dataset lớnKéo quá nhiều dữ liệu về 1 máy driverGiữ dữ liệu ở cluster, chỉ lấy sample nhỏ; xuất kết quả ra storage thay vì collect toàn bộ
Executor OOMTask fail ở stage join/groupBy, spill tăng mạnhSkew nặng, partition quá to, memory per task không đủGiảm skew/shuffle, tune partition size, và cấu hình memory + overhead hợp lý theo YARN/K8s
⚠️ Anti-pattern: bàn về Spark performance mà không định lượng shuffle cost. Đây là sai lầm phổ biến nhất ở tutorial beginner.

5) Narrow vs wide trực giác nhanh

PySpark intuition
# Narrow: dữ liệu ở lại trong partition hiện tại
df1 = df.select("user_id", "event_time").filter("event_time is not null")

# Wide: cần gom/chia lại dữ liệu theo key
df2 = df.groupBy("user_id").count()  # shuffle
df3 = df.join(dim_users, "user_id")  # thường shuffle

6. Bài Tập Thực Hành▶
Shuffle diagnosis task · Vẽ stage boundary

Cho pipeline: read parquet → filter country='VN' → select 6 cột → groupBy(user_id).agg(sum(revenue)) → join dim_users → write parquet. Hãy đánh dấu bước nào có khả năng tạo shuffle và giải thích.

  • filter, select thường là narrow, chưa cần shuffle.
  • groupBy(user_id) tạo shuffle để gom key.
  • join dim_users có thể shuffle (trừ khi broadcast được dim).
  • Tư duy ưu tiên: giảm dữ liệu trước groupBy/join để giảm shuffle volume.

Data Model & Execution Mindset
RDD vs DataFrame vs Dataset, transformations/actions, lazy evaluation, lineage
1
1. Menu Tổng Thể▼
Chủ đềÝ chínhQuy tắc thực chiến
RDDLow-level, flexible nhưng verboseDùng khi cần control đặc biệt

2
2. Mục Đích Module▼
  • Hiểu vì sao DataFrame là lựa chọn mặc định cho Data Engineer.
  • Biết cách đọc lineage và dự đoán chi phí trước khi chạy action.
  • Tránh 2 anti-pattern nguy hiểm: collect() lớn và UDF mù quáng.
3
3. Giải Thích Chi Tiết▼

1) RDD vs DataFrame vs Dataset

Tiêu chíRDDDataFrameDataset (Scala)
SchemaKhông bắt buộcCó schema rõTyped schema
OptimizerHạn chếCatalyst + TungstenCatalyst + typed safety
Dễ dùng cho DETrung bìnhCaoCao khi team Scala

2) DataFrame default choice cho DE

  • API declarative, dễ review và dễ tối ưu.
  • Built-in functions phong phú, giảm cần UDF.
  • Catalyst optimizer tự chọn physical plan tốt hơn code low-level thủ công trong đa số case.
  • Khi đọc Parquet/ORC, Spark có thể tận dụng predicate pushdown và vectorized reader để tránh đọc dữ liệu thừa, đặc biệt hiệu quả với bảng lớn.

3) Transformations vs Actions + lazy evaluation

PySpark example
from pyspark.sql import functions as F

prepared = (
    raw_df
    .filter(F.col("event_date") >= F.lit("2026-01-01"))
    .select("user_id", "event_date", "amount")
    .withColumn("amount_usd", F.col("amount") / F.lit(25000))
)

# Chưa chạy gì đến đây (lazy)
prepared.write.mode("overwrite").partitionBy("event_date").parquet("/curated/orders")
# Action write mới kích hoạt DAG execution

4) Lineage và debugging

Lineage cho bạn biết dữ liệu đã đi qua những bước nào. Khi job sai kết quả hoặc chậm, hãy quay lại lineage để xem bước nào phình dữ liệu hoặc tạo shuffle không cần thiết.

5) Anti-patterns phải tránh

⚠️ collect() dataset lớn lên driver có thể làm driver OOM. Chỉ collect sample nhỏ hoặc metric nhỏ.
⚠️ Dùng Python UDF bừa bãi làm giảm khả năng optimize/vectorize. Ưu tiên built-in functions trước.
Bad vs better
# Bad: collect full table
rows = df.collect()

# Better: lấy sample nhỏ để inspect
sample_rows = df.limit(20).toPandas()

# Better cho transform: built-in thay vì UDF khi có thể
df2 = df.withColumn("domain", F.regexp_extract("email", "@(.+)$", 1))

Execution mindset task · Dự đoán action

Đọc đoạn code Spark có 6 transformations và 2 actions (count(), write()). Hãy chỉ ra thời điểm cluster thực thi, và action nào có thể trigger full scan.

Transformations chưa chạy ngay. count() và write() đều là action, đều có thể đọc toàn bộ lineage nếu không cache hoặc không có pruning hợp lệ.

Join Strategies & Shuffle Optimization
Broadcast Join, Sort-Merge Join, Shuffle Hash Join và quyết định theo kích thước
1
1. Menu Tổng Thể▼
Join typeKhi nào phù hợpLưu ý
Broadcast JoinDim nhỏ, fact lớnQuan sát spark.sql.autoBroadcastJoinThreshold (mặc định thường 10MB)
Sort-Merge JoinHai bảng lớn, key join rõChi phí sort + shuffle cao
Shuffle Hash JoinMột số cluster/config cho phépNhạy với memory phân vùng
Skew handlingKey lệch nặngCần salting/split strategy hoặc AQE skew optimization
2
2. Mục Đích Module▼
  • Ra quyết định join theo data size, key distribution, và resource.
  • Giảm shuffle bằng pre-aggregation, projection cột cần thiết, và partition strategy hợp lý.
  • Nhận diện anti-pattern “join bảng rộng mù quáng”.
3
3. Giải Thích Chi Tiết▼

1) Broadcast Join

Dùng khi một bảng đủ nhỏ để gửi đến tất cả executors. Fact giữ nguyên partition, dim được broadcast nên giảm shuffle lớn ở fact side.

Thực tế production cần kiểm tra spark.sql.autoBroadcastJoinThreshold (thường mặc định ~10MB). Nếu dim vượt ngưỡng, Spark có thể không broadcast trừ khi bạn hint hoặc tune config.

Broadcast join example
from pyspark.sql import functions as F

fact = spark.read.parquet("/raw/orders")           # ~2TB/day, ~10B rows
dim  = spark.read.parquet("/raw/customers_dim")    # ~8MB sau khi select cột cần thiết

result = fact.join(F.broadcast(dim.select("customer_id", "segment")), "customer_id", "left")

2) Sort-Merge Join

Khi cả hai bảng đều lớn, Spark thường chọn sort-merge join. Nó cần sắp xếp key ở hai phía và shuffle dữ liệu theo key join.

3) Shuffle Hash Join

Có thể xuất hiện tùy plan/config. Ưu điểm là bỏ bớt sort trong một số tình huống, nhưng rủi ro memory cao nếu partition không cân bằng.

4) Giảm shuffle trong thực tế

  • Pre-aggregation: giảm số dòng trước join.
  • Select cột cần thiết: giảm width của row.
  • Filter sớm: cắt bớt dữ liệu đầu vào join.
Pre-aggregation before join
daily_orders = (
    fact_orders
    .filter("order_date >= '2026-01-01'")
    .groupBy("customer_id", "order_date")
    .agg(F.sum("amount").alias("daily_amount"))
)

joined = daily_orders.join(F.broadcast(dim_customers.select("customer_id", "region")), "customer_id")

5) AQE trong Spark 3.x (bắt buộc phải biết)

Adaptive Query Execution (AQE) cho phép Spark tối ưu lại plan ở runtime: coalesce shuffle partitions, xử lý skew join tốt hơn, và trong một số case có thể đổi strategy join khi thống kê runtime cho thấy bảng đủ nhỏ để broadcast.

📌 Khi tune Spark 3.x, luôn kiểm tra AQE đang bật và quan sát plan thực tế trong SQL tab. AQE rất mạnh nhưng không thay thế hoàn toàn data modeling và join design.

6) Skewed join và salting code mẫu

Nếu một key chiếm quá nhiều dòng (ví dụ country='US' chiếm 70%), một vài tasks sẽ chạy lâu bất thường. Đây là skew. Cách xử lý gồm salting key hoặc tách path xử lý key nóng.

Manual salting cho skew join
from pyspark.sql import functions as F

salt_buckets = 16

fact_salted = (
    fact_orders                                     # ~2TB/day
    .withColumn("salt", (F.rand(seed=42) * salt_buckets).cast("int"))
)

dim_expanded = (
    dim_customers                                   # bảng nhỏ hơn, có thể nhân bản theo salt
    .withColumn("salt", F.explode(F.array(*[F.lit(i) for i in range(salt_buckets)])))
)

joined = fact_salted.join(
    dim_expanded,
    on=["customer_id", "salt"],
    how="left"
).drop("salt")
⚠️ Anti-pattern: join hai bảng cực rộng (nhiều cột) rồi mới filter. Bạn đang shuffle cả dữ liệu không dùng tới.

6. Bài Tập Thực Hành▶
Join strategy decision task

Bảng fact: 900GB, dim_product: 120MB, dim_store: 2.5GB. Hãy đề xuất strategy cho từng join và lý do.

  • fact ⨝ dim_product: broadcast join hợp lý (120MB, cần kiểm tra threshold/cluster memory).
  • fact ⨝ dim_store: thường sort-merge/shuffle hash tùy plan vì 2.5GB khó broadcast an toàn.
  • Trước cả hai join: filter cột và pre-aggregate để giảm shuffle volume.

Partitioning, Bucketing, sortWithinPartitions & File Layout
Thiết kế layout dữ liệu để giảm chi phí đọc/ghi và tối ưu downstream query
1
1. Menu Tổng Thể▼
Chủ đềNội dungMục tiêu
repartition vs coalesceTăng/giảm partition chủ độngKiểm soát parallelism và small files
partitionByLayout file khi writeGiúp pruning downstream
bucketingHash bucket theo keyTối ưu join lặp lại
sortWithinPartitionsSort cục bộ theo partitionGiảm global sort cost
2
2. Mục Đích Module▼
  • Thiết kế partition column theo truy vấn thật, không theo cảm giác.
  • Hiểu trade-off bucketing: lợi ích join lặp lại vs chi phí maintain.
  • Biết tránh bẫy high-cardinality partition và bùng nổ file nhỏ.
3
3. Giải Thích Chi Tiết▼

1) repartition vs coalesce

  • repartition(n, key): có shuffle, dùng khi cần phân phối lại dữ liệu rõ ràng.
  • coalesce(n): giảm partition với ít shuffle hơn, phù hợp khi giảm số file trước write.

2) partitionBy khi ghi file + bẫy mặc định 200 partitions

spark.sql.shuffle.partitions mặc định thường là 200. Đây không phải con số “đúng cho mọi workload”. Với data lớn (hàng trăm GB đến TB), để default mù quáng có thể gây task quá nặng hoặc quá nhiều file nhỏ tùy pipeline.

Heuristic đơn giản: nhắm kích thước output partition khoảng 128MB-256MB rồi suy ra số partition ban đầu, sau đó đo lại qua Spark UI.

Write with partitionBy
(
  transformed_df                                  # ~2.8TB / 18 tháng dữ liệu giao dịch
  .repartition(1200, "event_date")               # ví dụ tune theo target 128-256MB/partition
  .write
  .mode("overwrite")
  .partitionBy("event_date")
  .parquet("s3://de-curated/events")
)

3) Chọn partition column như thế nào?

Tiêu chíNênKhông nên
CardinalityThấp-vừa (vd: date, region)Rất cao (user_id, transaction_id)
Query patternHay filter theo cột đóHiếm khi filter
GrowthCó thể kiểm soát số partition theo thời gianBùng nổ thư mục/file

4) Bucketing và sortWithinPartitions

Bucketing hữu ích khi join lặp lại trên cùng key trong workload ổn định. Đổi lại, pipeline ghi dữ liệu phức tạp hơn và chi phí vận hành tăng.

sortWithinPartitions vs sort global
from pyspark.sql import functions as F

optimized = (
    df.repartition(80, F.col("event_date"))
      .sortWithinPartitions(F.col("event_date"), F.col("host"))
)
# sortWithinPartitions: sort cục bộ, nhẹ hơn global sort nhiều trường hợp

5) File layout cho downstream engines

  • Giữ file size mục tiêu ổn định (ví dụ 128MB-512MB tùy platform).
  • Tránh quá nhiều small files gây overhead metadata và scan.
  • Đồng bộ partition strategy với cách BI/warehouse query thực tế.
  • Ưu tiên Parquet/ORC để tận dụng predicate pushdown và vectorized reader, giúp giảm bytes đọc từ storage.
  • Nếu dùng Delta Lake/Iceberg, lên kế hoạch compaction/OPTIMIZE định kỳ để kiểm soát small files dài hạn.
⚠️ Anti-pattern: partition theo cột high-cardinality mà không có nhu cầu filter tương ứng. Hậu quả: hàng triệu file nhỏ, query chậm hơn.

3. Giải Thích Chi Tiết▼

1) cache vs persist

  • cache() mặc định tương đương persist(MEMORY_AND_DISK) trong nhiều bối cảnh DataFrame API.
  • persist(level) cho phép chọn storage level chi tiết hơn.
  • Cache chỉ đáng khi dataframe được reuse nhiều lần trong cùng job/workflow.
Cache đúng ngữ cảnh
from pyspark import StorageLevel

base = spark.read.parquet("/raw/events").filter("event_date >= '2026-01-01'")
base = base.persist(StorageLevel.MEMORY_AND_DISK)

kpi1 = base.groupBy("country").count()
kpi2 = base.groupBy("device_type").agg({"revenue": "sum"})

kpi1.write.mode("overwrite").parquet("/tmp/kpi1")
kpi2.write.mode("overwrite").parquet("/tmp/kpi2")
base.unpersist()

2) Khi caching gây hại

  • Cache dataframe quá lớn nhưng chỉ dùng một lần.
  • Cache quá nhiều dataset khiến executor memory pressure tăng và spill nhiều hơn.
⚠️ Anti-pattern: cache mọi dataframe “cho chắc”. Đây là cách nhanh nhất để làm cluster chậm đi.

4) Bottleneck đọc như thế nào?

  • Skew: vài task chạy lâu vượt trội phần còn lại.
  • Large shuffle: shuffle read/write tăng đột biến.
  • Spill: memory không đủ, spill to disk cao.
  • Slow stages: stage dài bất thường do join/sort nặng.
  • GC pressure: executor quá nhiều cores/RAM trong 1 process cũng có thể làm GC nặng; thực tế thường bắt đầu với 4-5 cores/executor rồi đo tiếp.

from pyspark.sql import DataFrame, functions as F

def enrich_orders(df: DataFrame) -> DataFrame:
    return (
        df.filter(F.col("amount") > 0)
          .withColumn("amount_bucket", F.when(F.col("amount") >= 1000, "HIGH").otherwise("LOW"))
          .select("order_id", "customer_id", "amount_bucket")
    )

2) Test với fake DataFrame

test_transform_functions.py
def test_enrich_orders(spark):
    input_df = spark.createDataFrame([
        ("o1", "c1", 1200),
        ("o2", "c2", 200),
        ("o3", "c3", -10)
    ], ["order_id", "customer_id", "amount"])

    actual = enrich_orders(input_df)
    rows = {(r.order_id, r.amount_bucket) for r in actual.collect()}

    assert rows == {("o1", "HIGH"), ("o2", "LOW")}
