2. Mục Đích Module ▼

📊 Fact Table là gì?

Fact table là bảng trung tâm trong Star Schema — nơi lưu trữ các sự kiện nghiệp vụ đã xảy ra. Mỗi row trong fact table đại diện cho 1 sự kiện có thể đo lường được.

  • Dimension table: lưu context (ai, cái gì, ở đâu, khi nào)
  • Fact table: lưu measurements (bao nhiêu, bao lâu, số lần)
  • Fact table kết nối với dimension tables qua foreign keys
  • Thường là bảng lớn nhất trong DWH — có thể hàng tỷ rows

🎯 Sau tuần này bạn có thể

  • Phân biệt và chọn đúng loại fact table cho từng use case
  • Viết query deduplication chuẩn production bằng ROW_NUMBER
  • Implement Microbatch deduplication cho data stream hourly
  • Thiết kế reduced fact table với bit array để tiết kiệm storage
  • Viết MERGE statement idempotent cho incremental load
  • Commit artifact homework lên GitHub

🌾 Nhắc lại: Grain của Fact Table

Grain = định nghĩa "1 row trong bảng này đại diện cho cái gì". Đây là quyết định quan trọng nhất khi thiết kế fact table.

grain examples
-- GRAIN TỐT — cụ thể, không mơ hồ:
"1 row = 1 giao dịch bán hàng từ 1 khách hàng tại 1 cửa hàng vào 1 thời điểm"
"1 row = 1 page view từ 1 user trên 1 device vào 1 session"
"1 row = snapshot tồn kho của 1 sản phẩm tại 1 kho vào 1 ngày"

-- GRAIN TỆ — quá mơ hồ:
"1 row = thông tin về orders" ← orders gì? của ai? tổng hay từng line?
"1 row = user activity"     ← activity loại nào? daily? hourly? per event?
⚠️ Nguyên tắc vàng: Xác định grain TRƯỚC KHI thiết kế bất kỳ cột nào. Grain sai → model sai → query sai → business report sai → mất niềm tin của stakeholder.

⚡ 1. Transaction Fact Table

Grain: 1 event xảy ra tại 1 thời điểm. Mỗi lần user mua hàng, click, login → 1 row mới. Không bao giờ update row cũ.

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL · transaction fact example
-- Grain: 1 row = 1 line item trong 1 đơn hàng
CREATE TABLE fact_order_items (
    order_id        VARCHAR(50)   NOT NULL,
    order_line_num  SMALLINT      NOT NULL,
    date_sk         INT           NOT NULL,
    customer_sk     INT           NOT NULL,
    product_sk      INT           NOT NULL,
    quantity        INT           NOT NULL,
    unit_price      NUMERIC(10,2) NOT NULL,
    net_amount      NUMERIC(12,2) NOT NULL,
    loaded_at       TIMESTAMP     DEFAULT NOW()
);

-- ✅ Chỉ INSERT, không UPDATE bao giờ
-- Query: doanh thu theo ngày và category
SELECT
    d.full_date,
    p.category,
    SUM(f.net_amount)  AS revenue,
    COUNT(*)           AS transactions
FROM fact_order_items f
JOIN dim_date d     ON f.date_sk = d.date_sk
JOIN dim_product p  ON f.product_sk = p.product_sk
GROUP BY d.full_date, p.category
ORDER BY d.full_date;

📸 2. Periodic Snapshot Fact Table

Grain: 1 entity tại 1 thời điểm cố định (cuối ngày, cuối tuần, cuối tháng). INSERT thêm 1 row mỗi period — không xóa row cũ. Dùng cho semi-additive facts.

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL · periodic snapshot
-- Grain: 1 row = balance của 1 tài khoản vào cuối 1 ngày
CREATE TABLE fact_account_daily_snapshot (
    snapshot_date   DATE          NOT NULL,
    account_sk      INT           NOT NULL,
    customer_sk     INT           NOT NULL,
    closing_balance NUMERIC(15,2) NOT NULL, -- semi-additive!
    transaction_cnt INT           NOT NULL, -- additive
    debit_total     NUMERIC(15,2) NOT NULL, -- additive
    credit_total    NUMERIC(15,2) NOT NULL, -- additive
    PRIMARY KEY (snapshot_date, account_sk)
);

-- ✅ Trend analysis: balance trung bình theo tháng
SELECT
    DATE_TRUNC('month', snapshot_date) AS month,
    AVG(closing_balance)               AS avg_balance,  -- AVG, không SUM!
    SUM(transaction_cnt)               AS total_txns    -- SUM OK
FROM fact_account_daily_snapshot
GROUP BY DATE_TRUNC('month', snapshot_date);

🔄 3. Accumulating Snapshot Fact Table

Grain: 1 row = 1 process lifecycle (đơn hàng từ lúc đặt đến lúc giao). Row được UPDATE mỗi khi có milestone mới. Phù hợp để track lead time, pipeline velocity.

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL · accumulating snapshot
-- Grain: 1 row = 1 đơn hàng từ lúc đặt → giao → hoàn thành
CREATE TABLE fact_order_fulfillment (
    order_sk            INT        NOT NULL PRIMARY KEY,
    order_id            VARCHAR(50) NOT NULL,
    customer_sk         INT,

    -- Milestone date keys (NULL nếu chưa đến milestone)
    ordered_date_sk     INT,    -- bao giờ cũng có
    confirmed_date_sk   INT,    -- NULL cho đến khi confirmed
    shipped_date_sk     INT,    -- NULL cho đến khi shipped
    delivered_date_sk   INT,    -- NULL cho đến khi delivered
    returned_date_sk    INT,    -- NULL nếu không return

    -- Lag facts (tính từ milestone trước)
    days_to_confirm     INT,    -- confirmed_date - ordered_date
    days_to_ship        INT,    -- shipped_date - confirmed_date
    days_to_deliver     INT,    -- delivered_date - shipped_date
    total_lead_days     INT,    -- delivered_date - ordered_date

    order_total         NUMERIC(12,2),
    current_status      VARCHAR(20),
    last_updated_at     TIMESTAMP DEFAULT NOW()
);

-- ✅ UPDATE khi có milestone mới
UPDATE fact_order_fulfillment
SET
    shipped_date_sk = :new_date_sk,
    days_to_ship    = :new_date_sk - confirmed_date_sk,
    current_status  = 'SHIPPED',
    last_updated_at = NOW()
WHERE order_id = :order_id;

-- Query: avg lead time theo tháng đặt hàng
SELECT
    d.year_month,
    AVG(total_lead_days) AS avg_lead_days,
    COUNT(*) FILTER (WHERE delivered_date_sk IS NOT NULL) AS delivered_cnt
FROM fact_order_fulfillment f
JOIN dim_date d ON f.ordered_date_sk = d.date_sk
GROUP BY d.year_month;

GROUP BY d.year_month;

⚡ Transaction Fact1 event = 1 row. INSERT only, không update. Immutable. Dùng cho audit trail, event history.
📸 Periodic Snapshot1 entity × 1 period = 1 row. INSERT mỗi kỳ. Dùng AVG cho semi-additive khi aggregate theo time.

🗑️ DELETE + INSERTIdempotent, row-level lock, xử lý late data bằng backfill. Wrap trong transaction. Tốt cho transaction fact.

🔬 Bài 1: Viết query phân tích trên host_activity_reduced

Từ bảng host_activity_reduced, viết query trả về: top 10 hosts theo tổng hits tháng, kèm average hits per active day, chỉ lấy host có số ngày active tối thiểu 15.

🐘 PostgreSQL
🔴 Oracle SQL
solution · top_hosts.sql
WITH expanded AS (
    SELECT
        host,
        month_start,
        v.hit,
        v.day_index
    FROM host_activity_reduced har
    CROSS JOIN LATERAL UNNEST(har.hit_array) WITH ORDINALITY AS v(hit, day_index)
)
SELECT
    host,
    month_start,
    SUM(hit) AS total_hits,
    COUNT(*) FILTER (WHERE hit > 0) AS active_days,
    ROUND(SUM(hit)::NUMERIC / NULLIF(COUNT(*) FILTER (WHERE hit > 0), 0), 1)
      AS avg_hits_per_active_day
FROM expanded
WHERE month_start = DATE_TRUNC('month', CURRENT_DATE)::DATE
GROUP BY host, month_start
HAVING COUNT(*) FILTER (WHERE hit > 0) >= 15
ORDER BY total_hits DESC
LIMIT 10;
solution · top_hosts_oracle.sql
WITH exploded AS (
    SELECT
        har.host,
        har.month_start,
        jt.hit
    FROM host_activity_reduced_oracle har,
         JSON_TABLE(
             har.hit_array_json,
             '$[*]' COLUMNS (
               hit NUMBER PATH '$'
             )
         ) jt
)
SELECT
    host,
    month_start,
    SUM(hit) AS total_hits,
    SUM(CASE WHEN hit > 0 THEN 1 ELSE 0 END) AS active_days,
    ROUND(
      SUM(hit) / NULLIF(SUM(CASE WHEN hit > 0 THEN 1 ELSE 0 END), 0),
      1
    ) AS avg_hits_per_active_day
FROM exploded
WHERE month_start = TRUNC(SYSDATE, 'MM')
GROUP BY host, month_start
HAVING SUM(CASE WHEN hit > 0 THEN 1 ELSE 0 END) >= 15
ORDER BY total_hits DESC
FETCH FIRST 10 ROWS ONLY;
