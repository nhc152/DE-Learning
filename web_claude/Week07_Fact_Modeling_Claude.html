2. Mục Đích Module ▼

📊 Fact Table là gì?

Fact table là bảng trung tâm trong Star Schema — nơi lưu trữ các sự kiện nghiệp vụ đã xảy ra. Mỗi row trong fact table đại diện cho 1 sự kiện có thể đo lường được.

  • Dimension table: lưu context (ai, cái gì, ở đâu, khi nào)
  • Fact table: lưu measurements (bao nhiêu, bao lâu, số lần)
  • Fact table kết nối với dimension tables qua foreign keys
  • Thường là bảng lớn nhất trong DWH — có thể hàng tỷ rows

🎯 Sau tuần này bạn có thể

  • Phân biệt và chọn đúng loại fact table cho từng use case
  • Viết query deduplication chuẩn production bằng ROW_NUMBER
  • Implement Microbatch deduplication cho data stream hourly
  • Thiết kế reduced fact table với bit array để tiết kiệm storage
  • Viết MERGE statement idempotent cho incremental load
  • Commit artifact homework lên GitHub

🌾 Nhắc lại: Grain của Fact Table

Grain = định nghĩa "1 row trong bảng này đại diện cho cái gì". Đây là quyết định quan trọng nhất khi thiết kế fact table.

grain examples
-- GRAIN TỐT — cụ thể, không mơ hồ:
"1 row = 1 giao dịch bán hàng từ 1 khách hàng tại 1 cửa hàng vào 1 thời điểm"
"1 row = 1 page view từ 1 user trên 1 device vào 1 session"
"1 row = snapshot tồn kho của 1 sản phẩm tại 1 kho vào 1 ngày"

-- GRAIN TỆ — quá mơ hồ:
"1 row = thông tin về orders" ← orders gì? của ai? tổng hay từng line?
"1 row = user activity"     ← activity loại nào? daily? hourly? per event?
⚠️ Nguyên tắc vàng: Xác định grain TRƯỚC KHI thiết kế bất kỳ cột nào. Grain sai → model sai → query sai → business report sai → mất niềm tin của stakeholder.

🔬 Cấu trúc điển hình

PostgreSQL · fact_sales DDL
CREATE TABLE fact_sales (
    -- Surrogate key (optional, dùng cho SCD join)
    sale_sk         BIGSERIAL,

    -- Degenerate dimension key (từ source, không có dim table)
    order_id        VARCHAR(50)   NOT NULL,
    order_line_num  INT           NOT NULL,

    -- Foreign keys → dimension tables
    date_sk         INT           NOT NULL,   -- → dim_date
    customer_sk     INT           NOT NULL,   -- → dim_customer
    product_sk      INT           NOT NULL,   -- → dim_product
    store_sk        INT           NOT NULL,   -- → dim_store

    -- Additive facts (có thể SUM theo mọi dimension)
    quantity_sold   INT           NOT NULL,
    gross_amount    NUMERIC(12,2) NOT NULL,
    discount_amount NUMERIC(12,2) NOT NULL DEFAULT 0,
    net_amount      NUMERIC(12,2) NOT NULL,   -- gross - discount
    cost_amount     NUMERIC(12,2) NOT NULL,

    -- Semi-additive fact (chỉ SUM theo 1 số dimension)
    -- (không dùng ở đây, dùng trong snapshot tables)

    -- Non-additive fact (không SUM được — chỉ AVG, MIN, MAX)
    unit_price      NUMERIC(10,2) NOT NULL,
    discount_pct    NUMERIC(5,2),

    -- Audit columns
    loaded_at       TIMESTAMP     DEFAULT NOW()
);

3. Giải Thích Chi Tiết ▼

⚡ 1. Transaction Fact Table

Grain: 1 event xảy ra tại 1 thời điểm. Mỗi lần user mua hàng, click, login → 1 row mới. Không bao giờ update row cũ.

PostgreSQL · transaction fact example
-- Grain: 1 row = 1 line item trong 1 đơn hàng
CREATE TABLE fact_order_items (
    order_id        VARCHAR(50)   NOT NULL,
    order_line_num  SMALLINT      NOT NULL,
    date_sk         INT           NOT NULL,
    customer_sk     INT           NOT NULL,
    product_sk      INT           NOT NULL,
    quantity        INT           NOT NULL,
    unit_price      NUMERIC(10,2) NOT NULL,
    net_amount      NUMERIC(12,2) NOT NULL,
    loaded_at       TIMESTAMP     DEFAULT NOW()
);

-- ✅ Chỉ INSERT, không UPDATE bao giờ
-- Query: doanh thu theo ngày và category
SELECT
    d.full_date,
    p.category,
    SUM(f.net_amount)  AS revenue,
    COUNT(*)           AS transactions
FROM fact_order_items f
JOIN dim_date d     ON f.date_sk = d.date_sk
JOIN dim_product p  ON f.product_sk = p.product_sk
GROUP BY d.full_date, p.category
ORDER BY d.full_date;

📸 2. Periodic Snapshot Fact Table

Grain: 1 entity tại 1 thời điểm cố định (cuối ngày, cuối tuần, cuối tháng). INSERT thêm 1 row mỗi period — không xóa row cũ. Dùng cho semi-additive facts.

PostgreSQL · periodic snapshot
-- Grain: 1 row = balance của 1 tài khoản vào cuối 1 ngày
CREATE TABLE fact_account_daily_snapshot (
    snapshot_date   DATE          NOT NULL,
    account_sk      INT           NOT NULL,
    customer_sk     INT           NOT NULL,
    closing_balance NUMERIC(15,2) NOT NULL, -- semi-additive!
    transaction_cnt INT           NOT NULL, -- additive
    debit_total     NUMERIC(15,2) NOT NULL, -- additive
    credit_total    NUMERIC(15,2) NOT NULL, -- additive
    PRIMARY KEY (snapshot_date, account_sk)
);

-- ✅ Trend analysis: balance trung bình theo tháng
SELECT
    DATE_TRUNC('month', snapshot_date) AS month,
    AVG(closing_balance)               AS avg_balance,  -- AVG, không SUM!
    SUM(transaction_cnt)               AS total_txns    -- SUM OK
FROM fact_account_daily_snapshot
GROUP BY DATE_TRUNC('month', snapshot_date);

🔄 3. Accumulating Snapshot Fact Table

Grain: 1 row = 1 process lifecycle (đơn hàng từ lúc đặt đến lúc giao). Row được UPDATE mỗi khi có milestone mới. Phù hợp để track lead time, pipeline velocity.

PostgreSQL · accumulating snapshot
-- Grain: 1 row = 1 đơn hàng từ lúc đặt → giao → hoàn thành
CREATE TABLE fact_order_fulfillment (
    order_sk            INT        NOT NULL PRIMARY KEY,
    order_id            VARCHAR(50) NOT NULL,
    customer_sk         INT,

    -- Milestone date keys (NULL nếu chưa đến milestone)
    ordered_date_sk     INT,    -- bao giờ cũng có
    confirmed_date_sk   INT,    -- NULL cho đến khi confirmed
    shipped_date_sk     INT,    -- NULL cho đến khi shipped
    delivered_date_sk   INT,    -- NULL cho đến khi delivered
    returned_date_sk    INT,    -- NULL nếu không return

    -- Lag facts (tính từ milestone trước)
    days_to_confirm     INT,    -- confirmed_date - ordered_date
    days_to_ship        INT,    -- shipped_date - confirmed_date
    days_to_deliver     INT,    -- delivered_date - shipped_date
    total_lead_days     INT,    -- delivered_date - ordered_date

    order_total         NUMERIC(12,2),
    current_status      VARCHAR(20),
    last_updated_at     TIMESTAMP DEFAULT NOW()
);

-- ✅ UPDATE khi có milestone mới
UPDATE fact_order_fulfillment
SET
    shipped_date_sk = :new_date_sk,
    days_to_ship    = :new_date_sk - confirmed_date_sk,
    current_status  = 'SHIPPED',
    last_updated_at = NOW()
WHERE order_id = :order_id;

-- Query: avg lead time theo tháng đặt hàng
SELECT
    d.year_month,
    AVG(total_lead_days) AS avg_lead_days,
    COUNT(*) FILTER (WHERE delivered_date_sk IS NOT NULL) AS delivered_cnt
FROM fact_order_fulfillment f
JOIN dim_date d ON f.ordered_date_sk = d.date_sk
GROUP BY d.year_month;

5. Tổng Kết Module ▼
⚡ Transaction Fact1 event = 1 row. INSERT only, không update. Immutable. Dùng cho audit trail, event history.
📸 Periodic Snapshot1 entity × 1 period = 1 row. INSERT mỗi kỳ. Dùng AVG cho semi-additive khi aggregate theo time.
🔄 Accumulating Snapshot1 lifecycle = 1 row. UPDATE khi có milestone mới. Dùng cho pipeline, workflow, funnel tracking.
📅 Milestone Date KeysNULL cho đến khi milestone xảy ra. Lag facts tính từ milestone trước — đo bottleneck.
⚠️ Semi-additive Trapinventory_qty, balance: SUM theo entity OK, SUM theo time = overcount. Luôn dùng AVG hoặc end-of-period.
🔍 Chọn đúng loạiHỏi: "Cần lịch sử event?" → Transaction. "Cần trend theo time?" → Snapshot. "Cần track pipeline?" → Accumulating.

3. Giải Thích Chi Tiết ▼

🔴 Tại sao duplicate xảy ra?

  • Source system retry: API timeout → retry → 2 lần ghi cùng 1 event
  • Pipeline retry: Airflow task fail giữa chừng, chạy lại từ đầu → double insert
  • Multiple producers: Nhiều service cùng ghi event cho cùng 1 action
  • CDC (Change Data Capture): Binlog có thể deliver same event nhiều lần
  • Backfill overlap: Backfill historical data trùng với data đã load
⚠️ Nguyên tắc: Không bao giờ assume source data sạch. Luôn dedup trước khi load vào fact table production. Duplicate trong fact table = revenue metric sai = business quyết định sai.

✅ ROW_NUMBER() Deduplication Pattern

PostgreSQL · ROW_NUMBER dedup — production standard
-- Bảng staging có duplicate event_id
-- Lấy row "đúng nhất" theo business rule: mới nhất (updated_at DESC)

WITH deduped AS (
    SELECT
        *,
        ROW_NUMBER() OVER (
            PARTITION BY event_id          -- business key
            ORDER BY updated_at DESC,    -- lấy bản mới nhất
                     created_at  DESC    -- tiebreaker
        ) AS rn
    FROM stg_events
    WHERE event_date = '2024-01-15'       -- filter partition hôm nay
)
SELECT
    event_id, user_id, event_type,
    event_properties, occurred_at
FROM deduped
WHERE rn = 1;                             -- chỉ lấy row rank 1

-- Verify: số rows trước và sau dedup
SELECT
    COUNT(*) AS total_raw,
    COUNT(DISTINCT event_id) AS unique_events,
    COUNT(*) - COUNT(DISTINCT event_id) AS duplicate_count
FROM stg_events
WHERE event_date = '2024-01-15';

⚡ DISTINCT ON — Syntax ngắn hơn (PostgreSQL only)

PostgreSQL · DISTINCT ON
-- Tương đương ROW_NUMBER với partition + order
-- DISTINCT ON (key) + ORDER BY key, tiebreaker
SELECT DISTINCT ON (event_id)
    event_id, user_id, event_type, occurred_at
FROM stg_events
WHERE event_date = '2024-01-15'
ORDER BY
    event_id,
    updated_at DESC;   -- lấy bản mới nhất cho mỗi event_id

-- ⚠️ Giới hạn: ORDER BY phải bắt đầu bằng cùng columns trong DISTINCT ON
-- ROW_NUMBER linh hoạt hơn và portable hơn (Spark SQL, BigQuery, Snowflake)

🔄 Microbatch Deduplication — Pattern cho high-volume stream

Khi data volume lớn (hàng triệu events/ngày), dedup toàn bộ cùng lúc tốn RAM và slow. Microbatch chia nhỏ theo giờ, dedup từng batch, rồi merge.

PostgreSQL · Microbatch hourly dedup pattern
-- Bước 1: Tạo bảng hourly deduped (chạy mỗi giờ)
CREATE TABLE IF NOT EXISTS stg_events_hourly_deduped (
    event_id     VARCHAR(50) NOT NULL,
    user_id      BIGINT,
    event_type   VARCHAR(50),
    occurred_at  TIMESTAMP,
    event_hour   TIMESTAMP,           -- DATE_TRUNC('hour', occurred_at)
    PRIMARY KEY (event_id, event_hour) -- PK ngăn duplicate trong cùng giờ
);

-- Bước 2: INSERT deduped rows cho giờ hiện tại (upsert)
INSERT INTO stg_events_hourly_deduped
WITH hourly_deduped AS (
    SELECT
        *,
        DATE_TRUNC('hour', occurred_at) AS event_hour,
        ROW_NUMBER() OVER (
            PARTITION BY event_id, DATE_TRUNC('hour', occurred_at)
            ORDER BY updated_at DESC
        ) AS rn
    FROM raw_events
    WHERE occurred_at >= DATE_TRUNC('hour', NOW())
      AND occurred_at <  DATE_TRUNC('hour', NOW()) + INTERVAL '1 hour'
)
SELECT event_id, user_id, event_type, occurred_at, event_hour
FROM  hourly_deduped
WHERE rn = 1
ON CONFLICT (event_id, event_hour) DO UPDATE SET
    updated_at = NOW();   -- idempotent: chạy nhiều lần không tăng rows

-- Bước 3: Cross-hour dedup (1 event có thể span nhiều giờ do late arrival)
WITH cross_hour_deduped AS (
    SELECT
        *,
        ROW_NUMBER() OVER (
            PARTITION BY event_id
            ORDER BY occurred_at DESC
        ) AS rn
    FROM stg_events_hourly_deduped
    WHERE event_hour >= CURRENT_DATE  -- full day merge
)
SELECT * FROM cross_hour_deduped WHERE rn = 1;
✅ Microbatch reference: Pattern này dựa trên EcZachly/microbatch-hourly-deduped-tutorial — repo của Zach Wilson. Đây là chuẩn được dùng tại nhiều công ty DE lớn.

🔬 Bài 1: Viết dedup query production cho stg_web_events

Bảng stg_web_events có columns: event_id, user_id, page_url, event_type, occurred_at, ingested_at. Viết query dedup lấy row mới nhất (ingested_at lớn nhất) cho mỗi event_id. Bao gồm duplicate count report.

solution · web_events_dedup.sql
-- Step 1: Duplicate report
SELECT
    event_date,
    COUNT(*)                              AS total_raw,
    COUNT(DISTINCT event_id)               AS unique_events,
    COUNT(*) - COUNT(DISTINCT event_id)  AS duplicates
FROM stg_web_events
GROUP BY event_date
ORDER BY event_date DESC;

-- Step 2: Dedup CTE
WITH deduped AS (
    SELECT
        event_id, user_id, page_url,
        event_type, occurred_at, ingested_at,
        ROW_NUMBER() OVER (
            PARTITION BY event_id
            ORDER BY    ingested_at DESC
        ) AS rn
    FROM stg_web_events
    WHERE event_date = CURRENT_DATE
)
INSERT INTO fact_web_events
    (event_id, user_id, page_url, event_type, occurred_at)
SELECT
    event_id, user_id, page_url, event_type, occurred_at
FROM  deduped
WHERE rn = 1
ON CONFLICT (event_id) DO NOTHING;

🧮 Vấn đề: fact table activity quá lớn

Giả sử bạn track xem user có active trên website vào mỗi ngày không. Với 10M users × 365 ngày = 3.65 tỷ rows, mỗi row chỉ lưu TRUE/FALSE. Lãng phí cực kỳ.

PostgreSQL · naive approach vs reduced
-- ❌ NAIVE: 1 row per user per day — 3.65 tỷ rows cho 10M users × 365 ngày
CREATE TABLE fact_user_activity_naive (
    user_id     BIGINT,
    activity_date DATE,
    is_active   BOOLEAN,
    PRIMARY KEY (user_id, activity_date)
);
-- Storage: 10M × 365 × ~30 bytes ≈ 100 GB

-- ✅ REDUCED: 1 row per user per month — bit array cho 31 ngày
CREATE TABLE fact_user_activity_reduced (
    user_id       BIGINT        NOT NULL,
    month_start   DATE          NOT NULL,   -- first day of month
    active_days   BIT(31)       NOT NULL,   -- bit 1=active, 0=inactive
    active_count  SMALLINT      NOT NULL,   -- precomputed count of 1-bits
    PRIMARY KEY (user_id, month_start)
);
-- Storage: 10M × 12 × ~20 bytes ≈ 2.4 GB — tiết kiệm 40x!

-- Ý nghĩa của bit array:
-- active_days = '1010100110000000000000000000000' nghĩa là:
-- Ngày 1=active, ngày 2=inactive, ngày 3=active, ngày 4=inactive, ngày 5=active...
-- Bit thứ N tương ứng với ngày N trong tháng

📝 Build reduced fact từ activity data

PostgreSQL · build bit array per user per month
-- Tạo bit array cho tháng 2024-01
INSERT INTO fact_user_activity_reduced
WITH daily_activity AS (
    -- Step 1: Aggregate — user nào active vào ngày nào?
    SELECT
        user_id,
        DATE_TRUNC('month', event_date)  AS month_start,
        EXTRACT(DAY FROM event_date)::INT AS day_of_month
    FROM fact_web_events
    WHERE event_date >= '2024-01-01'
      AND event_date <  '2024-02-01'
    GROUP BY user_id, DATE_TRUNC('month', event_date), EXTRACT(DAY FROM event_date)
),
bit_builder AS (
    -- Step 2: Build bit string per user per month
    SELECT
        user_id,
        month_start,
        -- Tạo string 31 ký tự '0'/'1', rồi cast sang BIT(31)
        STRING_AGG(
            CASE WHEN da.day_of_month = d.day
                 THEN '1' ELSE '0'
            END,
            '' ORDER BY d.day
        )::BIT(31) AS active_days
    FROM (
        SELECT DISTINCT user_id, month_start FROM daily_activity
    ) users
    CROSS JOIN generate_series(1,31) d(day)
    LEFT JOIN  daily_activity da USING (user_id, month_start)
        AND da.day_of_month = d.day
    GROUP BY user_id, month_start
)
SELECT
    user_id,
    month_start,
    active_days,
    LENGTH(REPLACE(active_days::TEXT, '0', '')) AS active_count
FROM bit_builder
ON CONFLICT (user_id, month_start) DO UPDATE SET
    active_days  = EXCLUDED.active_days,
    active_count = EXCLUDED.active_count;

📦 Alternative: PostgreSQL ARRAY

PostgreSQL · array-based reduced fact
-- Lưu metric per day dưới dạng array — flexible hơn BIT
CREATE TABLE fact_host_activity_reduced (
    host            VARCHAR(255)      NOT NULL,
    month_start     DATE              NOT NULL,
    hit_array       BIGINT[]          NOT NULL,   -- hit count per day [1..31]
    unique_visitors INTEGER[]         NOT NULL,   -- unique visitors per day
    PRIMARY KEY (host, month_start)
);

-- Insert: aggregate hits per day → store as array
INSERT INTO fact_host_activity_reduced
SELECT
    host,
    DATE_TRUNC('month', event_date) AS month_start,
    ARRAY_AGG(daily_hits    ORDER BY day_of_month)  AS hit_array,
    ARRAY_AGG(unique_visits ORDER BY day_of_month)  AS unique_visitors
FROM (
    SELECT
        host,
        event_date,
        EXTRACT(DAY FROM event_date)::INT AS day_of_month,
        COUNT(*)               AS daily_hits,
        COUNT(DISTINCT user_id) AS unique_visits
    FROM fact_web_events
    WHERE event_date >= '2024-01-01'
    GROUP BY host, event_date
) daily
GROUP BY host, DATE_TRUNC('month', event_date)
ON CONFLICT (host, month_start) DO UPDATE SET
    hit_array       = EXCLUDED.hit_array,
    unique_visitors = EXCLUDED.unique_visitors;

-- Query: tổng hits tháng 1 của host 'example.com'
SELECT
    host,
    (
        SELECT SUM(v)
        FROM UNNEST(hit_array) v
    ) AS total_hits_jan
FROM fact_host_activity_reduced
WHERE host = 'example.com'
  AND month_start = '2024-01-01';

3️⃣ MERGE cho Accumulating Snapshot

PostgreSQL 15+ · MERGE statement
-- MERGE: ANSI SQL standard — PostgreSQL 15+ support
-- Dùng cho accumulating snapshot khi có milestone update

MERGE INTO fact_order_fulfillment target
USING (
    -- Source: new/updated orders từ staging
    SELECT
        s.order_id,
        s.customer_sk,
        s.ordered_date_sk,
        s.confirmed_date_sk,
        s.shipped_date_sk,
        s.delivered_date_sk,
        s.order_total,
        s.current_status,
        -- Tính lag facts
        s.confirmed_date_sk - s.ordered_date_sk   AS days_to_confirm,
        s.shipped_date_sk   - s.confirmed_date_sk AS days_to_ship,
        s.delivered_date_sk - s.shipped_date_sk   AS days_to_deliver,
        s.delivered_date_sk - s.ordered_date_sk   AS total_lead_days
    FROM stg_orders s
    WHERE s.updated_date = CURRENT_DATE
) source
ON (target.order_id = source.order_id)
WHEN MATCHED THEN
    -- Order đã tồn tại → UPDATE milestones mới
    UPDATE SET
        confirmed_date_sk  = source.confirmed_date_sk,
        shipped_date_sk    = source.shipped_date_sk,
        delivered_date_sk  = source.delivered_date_sk,
        days_to_confirm    = source.days_to_confirm,
        days_to_ship       = source.days_to_ship,
        days_to_deliver    = source.days_to_deliver,
        total_lead_days    = source.total_lead_days,
        current_status     = source.current_status,
        last_updated_at    = NOW()
WHEN NOT MATCHED THEN
    -- Order mới → INSERT row đầu tiên
    INSERT (order_id, customer_sk, ordered_date_sk,
             confirmed_date_sk, order_total, current_status, last_updated_at)
    VALUES (source.order_id, source.customer_sk, source.ordered_date_sk,
             source.confirmed_date_sk, source.order_total,
             source.current_status, NOW());

        browser_type,
        CURRENT_DATE AS activity_date
    FROM stg_web_events
    WHERE event_date = CURRENT_DATE
)
SELECT
    COALESCE(t.user_id, y.user_id)           AS user_id,
    COALESCE(t.browser_type, y.browser_type) AS browser_type,
    CASE
        -- Nếu hôm nay active: append ngày hôm nay vào array
        WHEN t.activity_date IS NOT NULL
          AND y.device_activity_datelist IS NOT NULL
        THEN y.device_activity_datelist || ARRAY[t.activity_date]
        -- User mới: tạo array với ngày hôm nay
        WHEN t.activity_date IS NOT NULL
        THEN ARRAY[t.activity_date]
        -- User không active hôm nay: giữ array cũ
        ELSE y.device_activity_datelist
    END                                       AS device_activity_datelist,
    CURRENT_DATE                              AS date
FROM today_activity t
FULL OUTER JOIN yesterday y
    ON t.user_id      = y.user_id
   AND t.browser_type = y.browser_type
ON CONFLICT (user_id, browser_type, date) DO UPDATE SET
    device_activity_datelist = EXCLUDED.device_activity_datelist;

CREATE TABLE IF NOT EXISTS hosts_cumulated (
    host                    VARCHAR(255)  NOT NULL,
    host_activity_datelist  DATE[]        NOT NULL,
    date                    DATE          NOT NULL,
    PRIMARY KEY (host, date)
);

-- Incremental load pattern giống user_devices_cumulated
INSERT INTO hosts_cumulated
WITH
yesterday AS (
    SELECT host, host_activity_datelist, date
    FROM   hosts_cumulated
    WHERE  date = CURRENT_DATE - INTERVAL '1 day'
),
today_hosts AS (
    SELECT DISTINCT
        host,
        CURRENT_DATE AS activity_date
    FROM stg_web_events
    WHERE event_date = CURRENT_DATE
)
SELECT
    COALESCE(t.host, y.host)       AS host,
    CASE
        WHEN t.activity_date IS NOT NULL AND y.host_activity_datelist IS NOT NULL
        THEN y.host_activity_datelist || ARRAY[t.activity_date]
        WHEN t.activity_date IS NOT NULL
        THEN ARRAY[t.activity_date]
        ELSE y.host_activity_datelist
    END                            AS host_activity_datelist,
    CURRENT_DATE                   AS date
FROM today_hosts t
FULL OUTER JOIN yesterday y ON t.host = y.host
ON CONFLICT (host, date) DO UPDATE SET
    host_activity_datelist = EXCLUDED.host_activity_datelist;

📦 Artifact 3: host_activity_reduced

03_host_activity_reduced.sql
-- DDL: reduced fact với bit array (1 month per host)
CREATE TABLE IF NOT EXISTS host_activity_reduced (
    host            VARCHAR(255)  NOT NULL,
    month_start     DATE          NOT NULL,
    hit_array       BIGINT[]      NOT NULL,  -- hits per day [1..31]
    unique_visitors INTEGER[]     NOT NULL,  -- unique users per day
    total_hits      BIGINT        NOT NULL,  -- precomputed monthly total
    active_days     SMALLINT      NOT NULL,  -- days with any activity
    PRIMARY KEY (host, month_start)
);

-- Build từ hosts_cumulated + stg_web_events
INSERT INTO host_activity_reduced
WITH daily_metrics AS (
    SELECT
        host,
        event_date,
        EXTRACT(DAY FROM event_date)::INT  AS day_num,
        COUNT(*)                           AS daily_hits,
        COUNT(DISTINCT user_id)            AS daily_unique
    FROM stg_web_events
    WHERE event_date >= DATE_TRUNC('month', CURRENT_DATE)
      AND event_date <  DATE_TRUNC('month', CURRENT_DATE) + INTERVAL '1 month'
    GROUP BY host, event_date
),
aggregated AS (
    SELECT
        host,
        DATE_TRUNC('month', event_date)::DATE            AS month_start,
        ARRAY_AGG(daily_hits   ORDER BY day_num)         AS hit_array,
        ARRAY_AGG(daily_unique ORDER BY day_num)         AS unique_visitors,
        SUM(daily_hits)                                   AS total_hits,
        COUNT(DISTINCT event_date)                        AS active_days
    FROM daily_metrics
    GROUP BY host, DATE_TRUNC('month', event_date)
)
SELECT * FROM aggregated
ON CONFLICT (host, month_start) DO UPDATE SET
    hit_array       = EXCLUDED.hit_array,
    unique_visitors = EXCLUDED.unique_visitors,
    total_hits      = EXCLUDED.total_hits,
    active_days     = EXCLUDED.active_days;

-- Verify: top 5 hosts theo total_hits tháng này
SELECT host, month_start, total_hits, active_days
FROM   host_activity_reduced
WHERE  month_start = DATE_TRUNC('month', CURRENT_DATE)
ORDER BY total_hits DESC
LIMIT 5;

📋 README Design Decisions

README.md · week7_fact_modeling
# Week 7 — Fact Data Modeling

## Artifacts
1. user_devices_cumulated — cumulative device activity per user
2. hosts_cumulated — cumulative host activity
3. host_activity_reduced — monthly reduced fact với ARRAY

## Design Decisions
- **Cumulative với ARRAY**: Chọn ARRAY[] thay vì BIT vì
  date_list cần truy vấn cụ thể từng ngày, không chỉ count
- **FULL OUTER JOIN cho incremental**: Đảm bảo không mất
  user đã có history nhưng không active hôm nay
- **UPSERT thay vì INSERT-only**: Cumulated table cần
  rebuild được khi backfill historical data
- **Precompute total_hits**: Integer query nhanh hơn
  SUM(UNNEST(array)) khi full scan millions of hosts

## Kết quả chạy thử
- user_devices_cumulated: 45,230 users tracked
- hosts_cumulated: 1,892 unique hosts
- host_activity_reduced: 156 host-months, avg 18.3 active days
🔬
5. Bài Tập Thực Hành ▼
🔬 Bài 1: Viết query phân tích trên host_activity_reduced

Từ bảng host_activity_reduced, viết query trả về: top 10 hosts theo total_hits, kèm avg hits per active day = total_hits / active_days, filter chỉ hosts có active_days >= 15.

solution · top_hosts.sql
SELECT
    host,
    month_start,
    total_hits,
    active_days,
    ROUND(total_hits::NUMERIC / NULLIF(active_days, 0), 1)
        AS avg_hits_per_active_day
FROM host_activity_reduced
WHERE month_start  = DATE_TRUNC('month', CURRENT_DATE)
  AND active_days >= 15
ORDER BY total_hits DESC
LIMIT 10;
