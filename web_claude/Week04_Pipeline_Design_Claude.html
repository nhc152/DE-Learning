2. Tại Sao Tư Duy Pipeline Quan Trọng? ▼

Ba tuần đầu bạn đã có SQL nâng cao, Python ETL, và kỹ thuật data cleaning. Tuần 4 là bản lề — nơi bạn học cách tư duy như một Data Engineer thực thụ, không chỉ viết code chạy được mà còn hiểu tại sao pipeline phải an toàn, đáng tin cậy và có thể vận hành production.

🔴 Vấn đề phổ biến khi không có tư duy pipeline

  • Chạy lại pipeline bị duplicate data — không ai thiết kế idempotency từ đầu
  • Backfill 6 tháng dữ liệu mất cả tuần — vì chỉ biết full load
  • Pipeline fail lúc 3 giờ sáng, không ai biết — vì không có monitoring/alerting
  • Dashboard sai vì upstream table chậm 2 ngày — vì không có SLA và data contract
  • Không rollback được khi có bug — vì logic không idempotent

🟢 Những gì bạn sẽ có sau tuần này

  • Vẽ và giải thích được DAG cho bất kỳ pipeline ETL nào
  • Implement idempotency pattern trong SQL (PostgreSQL và Oracle)
  • Chọn đúng load strategy cho từng use case cụ thể
  • Thiết kế failure handling và monitoring checklist production-ready
  • Hiểu SLA và viết data contract đơn giản
💡 Connection to Week 5+: Tuần 5–6 bạn sẽ implement Dimensional Data Modeling — toàn bộ concept SCD Type 2, incremental load, backfill strategy đều xây trên nền tư duy pipeline tuần này.

1. DAG Là Gì? ▼

DAG (Directed Acyclic Graph) là đồ thị có hướng không có chu trình. Trong data engineering, DAG mô tả thứ tự thực thi các task trong một pipeline.

  • Node: mỗi task/step trong pipeline (extract, transform, load, check)
  • Edge (cạnh có hướng): dependency — "A phải xong trước khi B bắt đầu"
  • Acyclic (không có chu trình): không có task nào phụ thuộc vào chính nó

Parallel vs Serial Execution

LoạiKhi nào dùngVí dụ
Serial (tuần tự)Task B cần output của task AExtract → Transform → Load

2. DAG trong SQL — Biểu Diễn Dependency bằng CTE ▼

Trong SQL, CTE (Common Table Expression) chính là cách biểu diễn DAG tuyến tính — mỗi CTE là một node, phụ thuộc CTE trước đó.

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL 15+
pipeline_dag_cte.sql — PostgreSQL
-- DAG biểu diễn bằng CTE: mỗi CTE = 1 node trong pipeline
WITH

-- Node 1: Extract (raw source)
raw_orders AS (
    SELECT order_id, customer_id, amount, created_at, status
    FROM staging.orders
    WHERE created_at::date = '2024-01-15'
),

-- Node 2: Validate (filter hợp lệ)
validated_orders AS (
    SELECT *
    FROM raw_orders
    WHERE amount > 0
      AND customer_id IS NOT NULL
      AND status IN ('completed', 'shipped')
),

-- Node 3: Transform (aggregate)
revenue_daily AS (
    SELECT
        created_at::date     AS report_date,
        COUNT(*)            AS order_count,
        SUM(amount)         AS total_revenue,
        AVG(amount)         AS avg_order_value
    FROM validated_orders
    GROUP BY 1
),

-- Node 4: Quality Check (inline assertion)
quality_check AS (
    SELECT
        report_date,
        order_count,
        total_revenue,
        CASE
            WHEN order_count = 0      THEN 'FAIL: no orders'
            WHEN total_revenue <= 0  THEN 'FAIL: negative revenue'
            ELSE 'PASS'
        END AS check_status
    FROM revenue_daily
)

-- Node 5: Load (chỉ insert nếu PASS)
INSERT INTO marts.revenue_daily_fact
    (report_date, order_count, total_revenue, avg_order_value, loaded_at)
SELECT
    report_date, order_count, total_revenue, avg_order_value,
    NOW()
FROM quality_check
WHERE check_status = 'PASS';
Oracle SQL
pipeline_dag_cte.sql — Oracle
-- Oracle: CTE với WITH clause giống PostgreSQL

WITH

raw_orders AS (
    SELECT order_id, customer_id, amount, created_at, status
    FROM staging.orders
    WHERE TRUNC(created_at) = TO_DATE('2024-01-15', 'YYYY-MM-DD')
),

    SELECT *
    FROM raw_orders
    WHERE amount > 0
      AND customer_id IS NOT NULL
      AND status IN ('completed', 'shipped')

        TRUNC(created_at)       AS report_date,
        COUNT(*)               AS order_count,
        SUM(amount)            AS total_revenue,
        AVG(amount)            AS avg_order_value
    FROM validated_orders
    GROUP BY TRUNC(created_at)
),

quality_check AS (
    SELECT
        report_date, order_count, total_revenue,
        CASE
            WHEN order_count = 0      THEN 'FAIL: no orders'
            WHEN total_revenue <= 0  THEN 'FAIL: negative revenue'
            ELSE 'PASS'

)

    (report_date, order_count, total_revenue, avg_order_value, loaded_at)
SELECT
    report_date, order_count, total_revenue, avg_order_value,
    SYSDATE   -- Oracle dùng SYSDATE thay NOW()
FROM quality_check
WHERE check_status = 'PASS';

1. Idempotency Là Gì? ▼

Idempotency: Chạy pipeline N lần với cùng input luôn cho ra cùng output — không duplicate, không thiếu, không lỗi.

Công thức: f(f(x)) = f(x) — áp dụng hàm nhiều lần cho kết quả giống áp dụng 1 lần.

❌ Ví dụ KHÔNG idempotent

anti_pattern — KHÔNG dùng
-- Chạy lần 1 vào ngày 2024-01-15: INSERT 1000 rows ✓
-- Scheduler fail, chạy lại lần 2: INSERT thêm 1000 rows ❌ DUPLICATE!
INSERT INTO marts.revenue_daily
SELECT report_date, SUM(amount)
FROM staging.orders
WHERE report_date = '2024-01-15'
GROUP BY report_date;

✅ Pattern 1: DELETE + INSERT (PostgreSQL & Oracle)

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL
idempotent_delete_insert.sql — PostgreSQL
-- Trong 1 transaction: xóa ngày cũ trước, rồi insert mới
BEGIN;

DELETE FROM marts.revenue_daily
WHERE report_date = '2024-01-15';

INSERT INTO marts.revenue_daily (report_date, order_count, total_revenue)
SELECT
    created_at::date,
    COUNT(*),
    SUM(amount)
FROM staging.orders
WHERE created_at::date = '2024-01-15'
  AND status = 'completed'
GROUP BY 1;

COMMIT;
-- Chạy 10 lần vẫn cho kết quả y hệt → idempotent ✓
Oracle SQL
idempotent_delete_insert.sql — Oracle
-- Oracle: transaction tự động, dùng COMMIT/ROLLBACK tường minh
DELETE FROM marts.revenue_daily
WHERE report_date = TO_DATE('2024-01-15', 'YYYY-MM-DD');

INSERT INTO marts.revenue_daily (report_date, order_count, total_revenue)
SELECT
    TRUNC(created_at)   AS report_date,
    COUNT(*)            AS order_count,
    SUM(amount)         AS total_revenue
FROM staging.orders
WHERE TRUNC(created_at) = TO_DATE('2024-01-15', 'YYYY-MM-DD')
  AND status = 'completed'
GROUP BY TRUNC(created_at);

COMMIT;

✅ Pattern 2: INSERT ON CONFLICT / MERGE (UPSERT)

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL 9.5+
upsert_on_conflict.sql — PostgreSQL
-- Dùng UNIQUE constraint trên report_date
-- INSERT nếu chưa có, UPDATE nếu đã có
INSERT INTO marts.revenue_daily
    (report_date, order_count, total_revenue, updated_at)
SELECT
    created_at::date,
    COUNT(*),
    SUM(amount),
    NOW()
FROM staging.orders
WHERE created_at::date = '2024-01-15'
GROUP BY 1
ON CONFLICT (report_date) DO UPDATE SET
    order_count   = EXCLUDED.order_count,
    total_revenue = EXCLUDED.total_revenue,
    updated_at    = NOW();
-- Idempotent: chạy N lần luôn cho cùng kết quả ✓
Oracle SQL — MERGE
merge_upsert.sql — Oracle
-- Oracle dùng MERGE thay ON CONFLICT DO UPDATE
MERGE INTO marts.revenue_daily tgt
USING (
    SELECT
        TRUNC(created_at)  AS report_date,
        COUNT(*)           AS order_count,
        SUM(amount)        AS total_revenue
    FROM staging.orders
    WHERE TRUNC(created_at) = TO_DATE('2024-01-15', 'YYYY-MM-DD')
    GROUP BY TRUNC(created_at)
) src
ON (tgt.report_date = src.report_date)
WHEN MATCHED THEN UPDATE SET
    tgt.order_count   = src.order_count,
    tgt.total_revenue = src.total_revenue,
    tgt.updated_at    = SYSDATE
WHEN NOT MATCHED THEN INSERT
    (report_date, order_count, total_revenue, updated_at)
VALUES
    (src.report_date, src.order_count, src.total_revenue, SYSDATE);

COMMIT;

2. Câu Hỏi Ôn Tập — Idempotency ▼
Định nghĩa chính xác nhất của idempotency trong pipeline là gì?
A. Pipeline chạy nhanh nhất có thể bất kể số lần trigger
B. Chạy N lần với cùng input luôn cho ra cùng output — không có side effect tích lũy
C. Pipeline tự động rollback khi fail để về trạng thái ban đầu
D. Pipeline chỉ chạy đúng 1 lần dù được trigger nhiều lần
Tại sao pattern DELETE + INSERT phải nằm trong cùng 1 transaction?
A. Vì DELETE và INSERT dùng chung bộ nhớ, không thể tách ra
B. Atomicity: nếu INSERT fail, DELETE cũng rollback — tránh mất data giữa chừng
C. Transaction giúp DELETE + INSERT chạy nhanh hơn
D. Vì PostgreSQL yêu cầu DELETE và INSERT phải cùng transaction
Oracle MERGE khác với PostgreSQL INSERT ... ON CONFLICT như thế nào về cú pháp?
A. Oracle MERGE chỉ INSERT, không UPDATE được
B. Oracle MERGE cũng dùng ON CONFLICT nhưng viết khác chút
C. Oracle dùng MERGE INTO ... USING ... ON ... WHEN MATCHED/NOT MATCHED; PostgreSQL dùng INSERT ... ON CONFLICT DO UPDATE
D. Oracle MERGE chỉ dùng được cho UPDATE, còn INSERT phải dùng lệnh riêng
Khi nào nên dùng UPSERT (ON CONFLICT/MERGE) thay vì DELETE + INSERT?
A. Luôn dùng UPSERT vì nhanh hơn DELETE + INSERT
B. UPSERT khi update row-by-row hoặc giữ nguyên row không thay đổi; DELETE+INSERT khi replace cả partition
C. UPSERT chỉ dùng khi table có primary key, DELETE+INSERT khi không có
D. DELETE+INSERT chỉ dùng trong transaction; UPSERT không cần transaction
Trong PostgreSQL ON CONFLICT, từ khóa EXCLUDED đề cập đến gì?
A. Các row đã bị DELETE trước đó
B. Row mới đang cố INSERT (bị conflict) — dùng để lấy giá trị mới trong DO UPDATE SET
C. Các cột NULL trong row mới
D. Tên của UNIQUE constraint gây ra conflict
Một pipeline load data vào bảng fact_sales. Sau khi load, team phát hiện source data bị sai và cần reprocess lại dữ liệu của ngày hôm qua. Pipeline có idempotency thì xử lý như thế nào?
A. Phải truncate toàn bộ bảng rồi reprocess tất cả data từ đầu
B. Trigger lại pipeline cho ngày hôm qua — idempotency đảm bảo xóa data cũ và insert lại đúng
C. Phải manual fix từng row bị sai trong database
D. Chạy thêm 1 lần incremental append để override data cũ
Trong Oracle MERGE, nếu một row trong source KHÔNG tồn tại trong target, mệnh đề nào được thực thi?
A. WHEN MATCHED THEN UPDATE
B. WHEN NOT MATCHED THEN INSERT
C. ON condition
D. WHEN NOT MATCHED BY SOURCE THEN DELETE
Tại sao NOT idempotent pipeline đặc biệt nguy hiểm hơn các loại bug khác trong data engineering?
A. Vì nó làm pipeline chạy chậm hơn 10x
B. Data duplicate tích lũy âm thầm, không có error rõ ràng — dashboard sai mà không ai biết cho đến khi có incident
C. Vì nó tiêu tốn storage gấp đôi ngay lần chạy đầu tiên
D. Vì nó làm crash database ngay lần chạy thứ 2
Để làm pipeline idempotent theo partition ngày, điều kiện WHERE trong DELETE phải là gì?
A. Không cần WHERE, DELETE toàn bộ rồi INSERT lại từ đầu
B. WHERE report_date = :target_date — chỉ xóa đúng partition ngày đang process
C. WHERE 1=1 để xóa hết rồi INSERT lại tất cả
D. WHERE created_at > NOW() để chỉ xóa data mới nhất
Trong PostgreSQL, nếu INSERT ... ON CONFLICT DO NOTHING được dùng, điều gì xảy ra khi có conflict?
A. Raise lỗi như INSERT thông thường
B. Bỏ qua row conflict hoàn toàn — không INSERT, không UPDATE, không lỗi
C. Tự động UPDATE row cũ với giá trị mới
D. Rollback toàn bộ transaction khi gặp conflict

-- Oracle: dùng MERGE cho incremental idempotent
-- Bước 1: Lấy watermark
SELECT NVL(MAX(last_loaded_at), TO_DATE('1970-01-01', 'YYYY-MM-DD'))
INTO v_last_watermark
FROM pipeline.checkpoints
WHERE pipeline_name = 'orders_incremental';

-- Bước 2: MERGE incremental
MERGE INTO marts.orders_fact tgt
USING (
    SELECT order_id, customer_id, amount, created_at
    FROM staging.orders
    WHERE created_at > v_last_watermark
      AND created_at <= SYSDATE - 5/1440  -- buffer 5 phút
) src
ON (tgt.order_id = src.order_id)
WHEN MATCHED THEN UPDATE SET
    tgt.amount    = src.amount,
    tgt.loaded_at = SYSDATE
WHEN NOT MATCHED THEN INSERT
    (order_id, customer_id, amount, created_at, loaded_at)
VALUES
    (src.order_id, src.customer_id, src.amount, src.created_at, SYSDATE);

-- Bước 3: Update checkpoint
MERGE INTO pipeline.checkpoints tgt
USING (SELECT 'orders_incremental' AS pipeline_name, SYSDATE AS ts FROM DUAL) src
ON (tgt.pipeline_name = src.pipeline_name)
WHEN MATCHED THEN UPDATE SET tgt.last_loaded_at = src.ts
WHEN NOT MATCHED THEN INSERT (pipeline_name, last_loaded_at) VALUES (src.pipeline_name, src.ts);

COMMIT;

Retry với Exponential Backoff

retry_pattern.py
import time, logging

def run_with_retry(func, max_retries=3, base_delay=30):
    """Exponential backoff: 30s → 60s → 120s"""
    for attempt in range(max_retries):
        try:
            return func()
        except TransientError as e:
            delay = base_delay * (2 ** attempt)
            logging.warning(f"Attempt {attempt+1} failed: {e}. Retry in {delay}s")
            time.sleep(delay)
        except DataQualityError as e:
            send_to_dead_letter_queue(e)   # Không retry data quality error
            raise
    raise PipelineFailedError("Max retries exceeded")

Khi một record không thể xử lý được (schema sai, validation fail), thay vì drop hoặc block pipeline, đưa vào DLQ để:

-- ① ROW COUNT CHECK: Phát hiện zero-row hoặc volume drop bất thường
WITH today_count AS (
    SELECT COUNT(*) AS cnt
    FROM marts.orders_fact
    WHERE created_at::date = CURRENT_DATE
),
avg_7d AS (
    SELECT AVG(daily_cnt) AS avg_cnt
    FROM (
        SELECT created_at::date, COUNT(*) AS daily_cnt
        FROM marts.orders_fact
        WHERE created_at::date BETWEEN CURRENT_DATE-7 AND CURRENT_DATE-1
        GROUP BY 1
    ) d
)
SELECT
    today_count.cnt,
    avg_7d.avg_cnt,
    CASE
        WHEN today_count.cnt = 0
             THEN 'FAIL: zero rows today'
        WHEN today_count.cnt < avg_7d.avg_cnt * 0.5
             THEN 'WARN: volume dropped >50% vs 7-day avg'
        ELSE 'PASS'
    END AS row_count_check
FROM today_count, avg_7d;

-- ② NULL CHECK: Phát hiện cột quan trọng có NULL
SELECT
    COUNT(*) AS total,
    COUNT(amount)      AS non_null_amount,
    COUNT(customer_id) AS non_null_customer,
    CASE
        WHEN COUNT(*) - COUNT(amount) > 0
             THEN FORMAT('FAIL: %s NULL amounts', COUNT(*)-COUNT(amount))
        ELSE 'PASS'
    END AS null_check
FROM marts.orders_fact
WHERE created_at::date = CURRENT_DATE;

-- ③ FRESHNESS CHECK: Data có được load đúng giờ không?
SELECT
    MAX(loaded_at) AS last_load,
    NOW() - MAX(loaded_at) AS staleness,
    CASE
        WHEN NOW() - MAX(loaded_at) > INTERVAL '26 hours'
             THEN 'FAIL: data is stale >26h'
        ELSE 'PASS'
    END AS freshness_check
FROM marts.orders_fact;

Bài 1 — Pipeline Idempotent với MERGE/UPSERT ▼
📝 Yêu cầu

Bảng staging.page_views có cột: view_id BIGINT, user_id BIGINT, page_url TEXT, viewed_at TIMESTAMP, session_id TEXT

Viết pipeline idempotent để aggregate thành marts.daily_page_views với: report_date, page_url, view_count, unique_users. Chạy nhiều lần phải cho cùng kết quả.

🐘 PostgreSQL
🔴 Oracle
PostgreSQL
solution_p1.sql
BEGIN;

-- Idempotent: xóa ngày target trước
DELETE FROM marts.daily_page_views
WHERE report_date = '2024-01-15';

INSERT INTO marts.daily_page_views
    (report_date, page_url, view_count, unique_users)
SELECT
    viewed_at::date        AS report_date,
    page_url,
    COUNT(*)               AS view_count,
    COUNT(DISTINCT user_id) AS unique_users
FROM staging.page_views
WHERE viewed_at::date = '2024-01-15'
GROUP BY 1, 2;

COMMIT;
Oracle
solution_p1_oracle.sql
DELETE FROM marts.daily_page_views
WHERE report_date = TO_DATE('2024-01-15','YYYY-MM-DD');

INSERT INTO marts.daily_page_views
    (report_date, page_url, view_count, unique_users)
SELECT
    TRUNC(viewed_at)       AS report_date,
    page_url,
    COUNT(*)               AS view_count,
    COUNT(DISTINCT user_id) AS unique_users
FROM staging.page_views
WHERE TRUNC(viewed_at) = TO_DATE('2024-01-15','YYYY-MM-DD')
GROUP BY TRUNC(viewed_at), page_url;
COMMIT;

Bài 2 — Incremental Load với Checkpoint ▼
📝 Yêu cầu

Viết incremental load cho bảng staging.events → marts.events_fact. Tạo bảng checkpoint và viết query: (1) lấy watermark, (2) load data mới, (3) update checkpoint. Xét late data với buffer 10 phút.

🐘 PostgreSQL
🔴 Oracle
PostgreSQL
solution_p2.sql
-- Step 1: Lấy watermark
SELECT COALESCE(MAX(last_loaded_at), '1970-01-01'::timestamp)
FROM pipeline.checkpoints WHERE pipeline_name = 'events_incremental';

-- Step 2: Load data mới với buffer 10 phút
INSERT INTO marts.events_fact
    (event_id, user_id, event_type, occurred_at, loaded_at)
SELECT event_id, user_id, event_type, occurred_at, NOW()
FROM staging.events
WHERE occurred_at > :last_watermark
  AND occurred_at <= NOW() - INTERVAL '10 minutes'
ON CONFLICT (event_id) DO UPDATE SET
    loaded_at = NOW();

-- Step 3: Update checkpoint
INSERT INTO pipeline.checkpoints (pipeline_name, last_loaded_at)
VALUES ('events_incremental', NOW() - INTERVAL '10 minutes')
ON CONFLICT (pipeline_name) DO UPDATE SET
    last_loaded_at = NOW() - INTERVAL '10 minutes';
Oracle
solution_p2_oracle.sql
-- Oracle: buffer 10 phút = 10/1440
MERGE INTO marts.events_fact tgt
USING (
    SELECT event_id, user_id, event_type, occurred_at
    FROM staging.events
    WHERE occurred_at > v_last_watermark
      AND occurred_at <= SYSDATE - 10/1440
) src
ON (tgt.event_id = src.event_id)
WHEN MATCHED THEN UPDATE SET tgt.loaded_at = SYSDATE
WHEN NOT MATCHED THEN INSERT
    (event_id, user_id, event_type, occurred_at, loaded_at)
VALUES
    (src.event_id, src.user_id, src.event_type, src.occurred_at, SYSDATE);
COMMIT;
