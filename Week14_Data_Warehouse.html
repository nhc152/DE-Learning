Tổng Quan: Từ PostgreSQL sang Modern Data Warehouse
Hiểu rõ vì sao PostgreSQL local không còn đủ khi analytics scale và concurrency tăng mạnh.

OLTP vs OLAP: khác nhau ở mục tiêu tối ưu

Tiêu chíOLTP (PostgreSQL app DB)OLAP (Cloud Warehouse)
Đơn vị công việcInsert/Update/Delete nhỏ, liên tụcScan/Aggregate trên dữ liệu lớn
Mẫu truy vấnLookup theo PK, transaction ngắnJoin nhiều bảng, GROUP BY theo chiều phân tích
Độ trễ mục tiêuMili-giây cho user transactionGiây cho phân tích, ưu tiên throughput

Rủi ro khi dùng saiApp DB nghẽn khi analyst chạy query nặngChi phí tăng nhanh nếu thiếu governance query

Vì sao PostgreSQL local bắt đầu hụt hơi?

  • Compute + storage không tách: query phân tích nặng có thể ảnh hưởng trực tiếp workload transaction.
  • Concurrency hạn chế: nhiều analyst cùng chạy dashboard khiến queue time và contention tăng rõ.
  • Ops burden cao: tuning index/vacuum/backfill/replication tiêu tốn effort lặp lại mỗi tuần.
  • Dữ liệu đa định dạng tăng nhanh: JSON/Parquet/event logs cần kiến trúc linh hoạt hơn row-store truyền thống.
⚠️ PostgreSQL không “kém”. Nó rất mạnh cho OLTP và data mart nhỏ. Vấn đề là dùng sai workload khi analytics đã vượt ngưỡng scale.

Đánh giá readiness migration

Bạn có PostgreSQL 1.5TB, 60 dashboard, SLA 8h sáng. Hãy lập bảng 3 nhóm chỉ số: hiệu năng, vận hành, chi phí và đưa quyết định có nên migrate trong quý tới hay chưa.

  • Hiệu năng: p95 query latency, queue time, freshness delay.
  • Vận hành: số incident/tháng, thời gian khắc phục, effort tuning.
  • Chi phí: compute + nhân sự ops + cơ hội mất quyết định kinh doanh do trễ dữ liệu.

Ngưỡng gợi ý: p95 > 30 giây + incident > 4/tháng + backlog tối ưu kéo dài > 6 tuần ⇒ nên lập kế hoạch migration.

Columnar storage: lợi thế thật sự nằm ở I/O và compression

Khía cạnhRow-basedColumnar

Partitioning: đúng key thì rẻ, sai key thì tốn

  • Nguyên tắc chọn: ưu tiên cột xuất hiện trong WHERE của 60-80% truy vấn business.
  • Mặc định an toàn: partition theo event_date hoặc order_date ở bảng fact lớn.
  • Tránh over-partition: partition quá nhỏ theo user_id/order_id gây nổ metadata và small files.

Clustering: tối ưu locality cho join/group phổ biến

Use caseClustering key gợi ýLưu ý
Dashboard theo quốc gia/nền tảngcountry, platformHữu ích khi truy vấn lặp lại theo chiều này
Join fact với dimension lớncustomer_id hoặc product_idTheo dõi skew để tránh cụm dữ liệu lệch
Truy vấn theo thời gian + domainevent_date, domainKết hợp partition theo ngày + cluster theo domain
partition_pruning_postgres.sql
SELECT country, SUM(revenue) AS total_revenue
FROM fct_orders
WHERE order_date BETWEEN DATE '2026-04-01' AND DATE '2026-04-07'
GROUP BY country;

-- Anti-pattern: thiếu filter theo partition key
SELECT country, SUM(revenue)
FROM fct_orders
GROUP BY country;
partition_pruning_oracle.sql
SELECT country, SUM(revenue) AS total_revenue
FROM fct_orders
WHERE order_date BETWEEN DATE '2026-04-01' AND DATE '2026-04-07'
GROUP BY country;

-- Anti-pattern tương tự nếu thiếu filter partition
SELECT country, SUM(revenue)
FROM fct_orders
GROUP BY country;
✅ Không có partition key đúng vĩnh viễn. Mỗi quý nên review top query theo scan bytes để điều chỉnh partition/clustering theo workload mới.

Schema-on-Write vs Schema-on-Read
Biết khi nào cần linh hoạt, khi nào cần nghiêm ngặt để bảo toàn chất lượng dữ liệu.
1
1. 📋 Menu Tổng Thể▼
Chiến lượcKhi dùngRủi ro
Schema-on-WriteCurated/Gold layer, BI chính thứcGiảm linh hoạt ingest
Schema-on-ReadRaw ingest đa định dạngLỗi kiểu dữ liệu phát hiện muộn
Kết hợpLakehouse nhiều lớpCần governance và contract rõ
2
2. 🎯 Mục Đích Module▼
  • Phân biệt đúng 2 chiến lược schema trong pipeline hiện đại.
  • Hiểu rủi ro schema drift và cách giảm thiểu bằng tests/contracts.
  • Thiết kế phân tầng dữ liệu hợp lý cho lakehouse.
3
3. 📖 Giải Thích Chi Tiết▼

Schema-on-Write vs Schema-on-Read: khác ở thời điểm kiểm soát chất lượng

Khía cạnhSchema-on-WriteSchema-on-Read
Thời điểm validateTrước khi ghi vào bảng chuẩn hóaLúc transform hoặc lúc truy vấn
Ưu điểmDữ liệu downstream ổn định, dễ tin cậyIngest nhanh, linh hoạt với nguồn thay đổi liên tục
Rủi roGiảm tốc độ onboard nguồn mớiLỗi schema/type phát hiện muộn nếu thiếu governance
Tầng phù hợpGold/serving layerRaw và một phần silver

Thiết kế thực tế theo Raw/Silver/Gold

  • Raw (schema-on-read): lưu payload gần nguyên bản để không mất tín hiệu nguồn.
  • Silver (hybrid): parse trường chính, chuẩn hóa timezone/type, tách records lỗi vào quarantine.
  • Gold (schema-on-write): khóa schema và kiểm soát chặt tests cho KPI chính thức.

Anti-pattern cần tránh

Anti-patternHậu quảCách sửa
Để tất cả mãi ở rawData swamp, BI không tin số liệuThiết kế silver/gold có owner rõ ràng
Ép schema cứng quá sớmOnboard nguồn mới chậmDùng hybrid: raw linh hoạt, gold chặt
Không có contract với producerSchema drift phá pipeline âm thầmVersion schema + test tự động + alert
dbt_schema_contract.yml
models:
  - name: fct_orders
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        tests: [not_null, unique]
      - name: order_amount
        tests:
          - dbt_expectations.expect_column_values_to_be_between:
              min_value: 0
⚠️ Schema-on-read không phải “không cần schema”. Nếu thiếu data contract và test gate, pipeline sẽ trả giá bằng incident production.

Lakehouse là logic kiến trúc, không phải buzzword

Lakehouse xuất phát từ nhu cầu thực tế: doanh nghiệp cần vừa lưu raw data đa định dạng chi phí thấp như data lake, vừa cần độ tin cậy/SQL semantics như warehouse để phục vụ BI và data product ổn định.

4 lớp kiến trúc lakehouse điển hình

LớpMục tiêuCông cụ/thực hành thường gặp
Storage foundationLưu dữ liệu mở, bền vững, rẻS3/GCS/ADLS + Parquet
Table semanticsACID, snapshot, schema evolutionIceberg/Delta/Hudi
Compute/processingTransform, quality gate, orchestrationSpark/Flink/dbt/Airflow
Serving/consumptionBI, semantic layer, data productsWarehouse engine/BI tool/API

Khi nào lakehouse thực sự hợp lý?

  • Có nhiều workload cùng khai thác dữ liệu: BI + ad-hoc + data science.
  • Cần tránh lock-in quá chặt vào một engine compute.
  • Cần quản trị thống nhất metadata/lineage qua nhiều domain dữ liệu.

Khi chưa nên dùng lakehouse đầy đủ

  • Dữ liệu nhỏ, chỉ có BI cơ bản: warehouse managed thuần có thể đơn giản và hiệu quả hơn.
  • Đội chưa có năng lực metadata/governance: triển khai lakehouse nửa vời dễ thành data swamp.

📌 Lakehouse mạnh khi bạn quản lý tốt metadata + data contracts. Nếu bỏ phần governance, kiến trúc sẽ mất lợi thế cốt lõi.

Định vị đúng Apache Iceberg

Iceberg là gì?Iceberg không phải gì?
Open table format quản lý metadata + snapshot cho bảng lớnKhông phải data warehouse độc lập
Lớp semantics giữa object storage và compute engineKhông thay thế Spark/Trino/warehouse engine
Giúp nhiều engine truy cập bảng nhất quánKhông tự tối ưu nếu thiếu compaction/governance

Luồng ghi dữ liệu trong Iceberg (nhìn theo vận hành)

  1. Engine ghi data files mới (thường Parquet).
  2. Tạo manifest + manifest list chứa thống kê file.
  3. Commit metadata mới thành snapshot bất biến.
  4. Catalog cập nhật pointer sang snapshot mới nhất.
iceberg_table_demo.sql
CREATE TABLE IF NOT EXISTS lakehouse.matches_bucketed (
  match_id STRING,
  playlist_id STRING,
  completion_date TIMESTAMP
)
USING iceberg
PARTITIONED BY (days(completion_date), bucket(16, match_id));

SELECT * FROM lakehouse.matches_bucketed VERSION AS OF 123456789;

ALTER TABLE lakehouse.matches_bucketed ADD COLUMN map_name STRING;

Cost control: đo đúng trước khi tối ưu

Nhóm chỉ sốNên theo dõiMục tiêu vận hành
Hiệu năng truy vấnp50/p95 latency, queue timeGiữ dashboard SLA ổn định
Chi phíScan bytes, credits/slot time theo domainDự báo và khóa budget theo tuần/tháng
Hiệu suất dữ liệuPruning ratio, bytes scanned/rows returnedPhát hiện query waste sớm

Playbook tối ưu theo thứ tự ưu tiên

  1. Giảm scan không cần thiết: bỏ SELECT *, filter partition key bắt buộc.
  2. Giảm data trung gian: filter trước khi join, pre-aggregate KPI lặp lại.
  3. Tối ưu layout: điều chỉnh partition/clustering, compaction định kỳ.
  4. Đặt guardrail: quota theo team, alert khi vượt ngưỡng scan/credit.

-- Nên làm: filter theo thời gian + chỉ lấy cột cần

-- Không nên: quét rộng + cột dư
