3. Giải Thích Chi Tiết ▼

OVER clause gồm 3 phần cốt lõi

  • PARTITION BY: chia dữ liệu thành các nhóm độc lập.
  • ORDER BY: thứ tự tính toán bên trong mỗi partition.
  • Frame: phạm vi hàng tham gia tính toán tại mỗi dòng.

ROWS vs RANGE — Bẫy mặc định

Tiêu chíROWSRANGE
Cơ sởVị trí vật lý của dòngGiá trị logic của ORDER BY
Tie (giá trị bằng nhau)Mỗi dòng được tính riêngTất cả dòng cùng giá trị được gộp chung
Kết quả với duplicate datesKhác nhau — tính tuần tựGiống nhau — tất cả tie cùng giá trị
Khuyến nghị✅ Dùng cho rolling metricsÍ dùng khi cần logic giá trị
⚠️ Default frame trap: Khi có ORDER BY mà không ghi rõ frame, mặc định là RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — không phải ROWS. Điều này có thể gây sai kết quả running total nếu có nhiều dòng cùng ngày. Luôn ghi rõ ROWS BETWEEN cho mọi aggregate window.

Dedup Latest Record — Pattern quan trọng nhất cho DE

Bảng staging thường chứa nhiều bản ghi cho cùng một entity (do retry, CDC multi-version, hoặc source gửi trùng). Để lấy bản ghi mới nhất, dùng ROW_NUMBER() PARTITION BY key ORDER BY updated_at DESC rồi lọc rn = 1.

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL

Cú pháp chuẩn ANSI SQL, chạy hoàn toàn giống nhau trên PostgreSQL và Oracle.

-- Dedup: giữ bản ghi mới nhất theo order_id
WITH deduped AS (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY updated_at DESC
    ) AS rn
  FROM stg_orders
)
SELECT order_id, customer_id, amount, updated_at
FROM deduped
WHERE rn = 1;

Streak Analysis — Gap and Island (Chuỗi liên tiếp)

Bài toán: tìm user hoạt động liên tiếp 7+ ngày. Kỹ thuật: gán mỗi dòng một island group bằng cách trừ ngày cho ROW_NUMBER. Các ngày liên tiếp sẽ có cùng giá trị group.

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL

PostgreSQL yêu cầu ép kiểu ::int rõ ràng khi thực hiện phép trừ giữa ngày (date) và kết quả của hàm ROW_NUMBER().

postgresql · streak_analysis.sql
-- Bước 1: Gán island group
WITH numbered AS (
  SELECT
    user_id,
    activity_date,
    activity_date - (ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY activity_date
    ))::int AS island_group
  FROM user_activity
),
-- Bước 2: Đếm số ngày liên tiếp mỗi streak
streaks AS (
  SELECT
    user_id,
    island_group,
    MIN(activity_date) AS streak_start,
    MAX(activity_date) AS streak_end,
    COUNT(*) AS streak_days
  FROM numbered
  GROUP BY user_id, island_group
)
-- Bước 3: Lọc user có streak >= 7 ngày
SELECT user_id, streak_start, streak_end, streak_days
FROM streaks
WHERE streak_days >= 7
ORDER BY streak_days DESC;

Bài 5 · Tìm user hoạt động 5+ ngày liên tiếp

Từ bảng user_activity(user_id, activity_date), tìm tất cả user có streak hoạt động liên tiếp >= 5 ngày. Trả về user_id, streak_start, streak_end, streak_days.

solution.sql
WITH numbered AS (
  SELECT user_id, activity_date,
    activity_date - (ROW_NUMBER() OVER (
      PARTITION BY user_id ORDER BY activity_date
    ))::int AS grp
  FROM user_activity
),
streaks AS (
  SELECT user_id, grp,
    MIN(activity_date) AS streak_start,
    MAX(activity_date) AS streak_end,
    COUNT(*) AS streak_days
  FROM numbered
  GROUP BY user_id, grp
)
SELECT user_id, streak_start, streak_end, streak_days
FROM streaks
WHERE streak_days >= 5
ORDER BY streak_days DESC;

Idempotent UPSERT

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL

PostgreSQL dùng ON CONFLICT (key) DO UPDATE để upsert. EXCLUDED tham chiếu giá trị mới từ INSERT.

postgresql · incremental_upsert.sql
-- Tránh duplicate khi chạy lại
INSERT INTO dw_orders (order_id, customer_id, amount, updated_at)
SELECT order_id, customer_id, amount, updated_at
FROM stg_orders_delta
ON CONFLICT (order_id)
DO UPDATE SET
  customer_id = EXCLUDED.customer_id,
  amount = EXCLUDED.amount,
  updated_at = EXCLUDED.updated_at
WHERE dw_orders.updated_at < EXCLUDED.updated_at;
Oracle SQL

Oracle dùng MERGE INTO ... USING ... ON thay cho ON CONFLICT. MERGE hỗ trợ cả MATCHED và NOT MATCHED trong một câu.

oracle sql · incremental_upsert.sql
-- Tránh duplicate khi chạy lại bằng MERGE
MERGE INTO dw_orders d
USING stg_orders_delta s
   ON (d.order_id = s.order_id)
 WHEN MATCHED THEN
   UPDATE SET
     d.customer_id = s.customer_id,
     d.amount = s.amount,
     d.updated_at = s.updated_at
   WHERE d.updated_at < s.updated_at
 WHEN NOT MATCHED THEN

Late Arriving Data — Dữ liệu về muộn

Trong thực tế, không phải tất cả record đều có updated_at đúng thời điểm. Mạng bị lag, retry queue giữ record lại, hay microservice ghi chậm đều khiến dữ liệu có updated_at nằm trước watermark mà chưa được load. Đây gọi là Late Arriving Data.

Nếu pipeline chỉ lấy WHERE updated_at > last_wm mà không có biên độ an toàn, các record muộn sẽ bị bỏ sót vĩnh viễn.

🐘 PostgreSQL
🔴 Oracle SQL
PostgreSQL

PostgreSQL dùng ON CONFLICT kết hợp CTE watermark. Cú pháp ::timestamp để cast kiểu.

postgresql · safe_watermark.sql
-- Lùi watermark 1 giờ để bắt Late Arriving Data

  SELECT COALESCE(MAX(updated_at), '1970-01-01'::timestamp)
    - INTERVAL '1 hour' AS safe_wm
  FROM dw_orders
)
INSERT INTO dw_orders (order_id, customer_id, amount, updated_at)
SELECT s.order_id, s.customer_id, s.amount, s.updated_at
FROM src_orders s
CROSS JOIN wm
WHERE s.updated_at > wm.safe_wm
ON CONFLICT (order_id)

Oracle SQL

Oracle dùng MERGE thay ON CONFLICT và TO_TIMESTAMP thay cast ::timestamp.

oracle sql · safe_watermark.sql
-- Lùi watermark 1 giờ để bắt Late Arriving Data
MERGE INTO dw_orders d
USING (
  SELECT s.*
  FROM src_orders s,
    (SELECT COALESCE(MAX(updated_at),
       TO_TIMESTAMP('1970-01-01', 'YYYY-MM-DD'))
       - INTERVAL '1' HOUR AS safe_wm
     FROM dw_orders) wm
  WHERE s.updated_at > wm.safe_wm
) s ON (d.order_id = s.order_id)
WHEN MATCHED THEN
  UPDATE SET d.amount = s.amount, d.updated_at = s.updated_at
  WHERE d.updated_at < s.updated_at
WHEN NOT MATCHED THEN
  INSERT (order_id, customer_id, amount, updated_at)
  VALUES (s.order_id, s.customer_id, s.amount, s.updated_at);

-- ✅ ĐÚNG: đổi thành range filter → index hoạt động bình thường
